view-results

The model is built — now read its footprint. view-results opens the Results panel over the canvas: the emissions over time (yearly bars and a cumulative curve) and the Impact repartition Sankey that shows where the CO₂e actually flows. Success isn't "a chart appeared" — it's understanding the number and trusting it: every value can be traced to its inputs (a formula, a calculus graph, a cited source), filtered by the angle you care about, and exported. Results open in place over a model that's already computable; the button stays disabled with the live reason until it is.

Faithful to the deployed app. Draws from the Results bar / Show results pill, the result-chart panel and its graph + Sankey partials, the source-table, and the calculated-attribute explanation views. Note: value derivations are tooltips, inline expanders and a new-tab calculus graph — not centered modals. Code-level pointers are collected for agents in Implementation pointers at the foot.

1

Results waits until the model computes

Both entry points — the bottom bar and the toolbar pill — gate on the model being computable. While it isn't, they're disabled and the tooltip is the live reason from validation, so the user reads exactly what's missing rather than a flat "complete your model." It's the canvas's disabled, never errored rule, applied to results.

Two synced buttonsThe bottom Results bar and the toolbar Show results pill always agree — one shared region rebuilds both, so they never drift.
Live reasonThe locked tooltip is the real validation text (missing usage pattern, a journey with no steps, …) — not a static string.
Unlock = one beatWhen the last missing piece lands, both buttons enable and a toast fires: "Your model is complete — results are now available." Remove it and they re-lock with "Results are no longer available."

Open Results — emissions over time

Results opens in place — a panel over the canvas, not a new page. It leads with the two time views: a stacked yearly bar (toggle to monthly) and a cumulative curve, both in the model's own display unit. The legend is split into Fabrication and Usage so the two halves of the lifecycle read at a glance; clicking a capsule isolates or hides a series.

In-place panelSlides up over the canvas (full-screen on mobile); a down-chevron / Back to model closes it. No URL change.

GranularityThe Year / Month select redraws the bar and rewrites its title (Yearly → Monthly CO2 emissions). Default is Year.

Impact repartition — where the CO₂e flows

The Sankey answers "where does it come from?" — flow width is share of emissions, from the whole system down to components. Analyse by chips choose the columns (total → countries → journeys → steps → hardware → component breakdown); the diagram re-renders on every change. The heavy repartition maths runs server-side; the browser only lays the diagram out, with hover tooltips carrying each node's value and share.

Analyse byChips toggle the columns shown left-to-right; only chips whose objects exist in the model appear. Default shows phase, journeys, steps, category & hardware.
RefineLifecycle phase (All / Manufacturing only / Usage only) and an Aggregation threshold that folds sub-threshold flows into "Other."
AdvancedExcluded types chips, a Show column-headers toggle, and a Label length cap.
Stack viewsAdd another analysis view appends an independent card; each card's settings persist across reloads and survive export / import.

4

Sources — provenance, editable and exportable

Numbers are only as trustworthy as their origins, so the Sources tab puts every input's provenance in one table: its value, unit, cited source and a confidence badge. Input rows are editable in place — fixing a source or confidence is a metadata-only change that doesn't recompute the model. Calculated values aren't sourced; they link to their (see graph) derivation instead.

One row per inputItem · Attribute of · Object type · Value · Unit · Source · Confidence · Comment.
Inline editThe ✎ opens a row editor for source / confidence / comment; saving doesn't reload the table or recompute.
Calculated rowsShow (see graph) — a link to the value's full calculus graph — rather than a source.
Export to xlsxThe button (labelled "Export to xslx" in the app) downloads a sources spreadsheet. The model itself exports as JSON from the toolbar — see save-and-load.

Every value carries its derivation

A footprint you can't interrogate is a footprint you can't trust. So every computed value can be traced to its inputs — but through lightweight surfaces, not a heavyweight modal: tooltips on the charts and Sankey, an inline formula expander in an object's Calculated attributes, the Sources (see graph) link, and the full calculus graph opened in a new tab.

Decisions taken

DecisionWhy / why-not-X
Results is disabled with the live validation reason — never offered then errored.Opening results on an incomplete model would compute nothing. The button gates on computability and the tooltip is the actual error (which usage pattern / journey step is missing), so the user reads the fix — disabled-instead-of-error, with specifics.
Two synced entry points (bottom bar + toolbar pill), one shared region.The bar is discoverable (swipe-up on mobile); the pill is always in reach. Both are rebuilt by a single region so their enabled/disabled state can never drift apart.
Results open in place over the canvas, not as a new page or URL.The model stays loaded behind the panel; closing is instant and keeps context. Consistent with the rest of the app's no-reload, no-push-state behaviour.
Charts are drawn client-side; the Sankey's repartition maths runs server-side.Chart.js builds the time charts from emissions data already in the page; the Sankey's heavy attribution is the library's, kept as the single source of modeling truth — the browser only lays it out and shows precomputed tooltips.
Sankey settings live in UI config, not in the model; views stack as independent cards.An analysis angle is a viewing choice, not a model fact — so it persists per card across reloads and survives export/import without touching the model. Stacking cards beats one over-configurable diagram for comparing angles side by side.
Derivations use lightweight surfaces (tooltips, inline expanders, a new tab) — not a modal.A glance wants a tooltip; the formula wants to stay next to the value; the full graph is large, so it's cached and opened out-of-line in a new tab. A heavyweight modal would interrupt the read and bloat the payload.
⌗ Implementation pointers for agents — routes, the result-charts bundle, the Sankey pipeline & the derivation views

Where this lives in code. Start with specs/architecture.md (render layer, OOB regions, interface_config / Sankey persistence). The gate & live reason mirror build-a-model's Results bar.

The gate (Phase A)
components/results_bar_button.html (#btn-open-panel-result) + components/show_results_toolbar_button.html (#show-results-toolbar-btn, label "Show results"), both gating on creation_constraints["__results__"]. Live reason from ModelWeb._build_creation_constraints → SystemValidationService.validate_for_computation. Lock/unlock toast: results_buttons OOB region (modeling_object_web.py, oob_regions.py) + CONSTRAINT_MESSAGES["__results__"] ("Your model is complete — results are now available" / "Results are no longer available").
Panel & charts (Phase B)
Route result-chart/ → result_chart (sets HX-Trigger-After-Settle: triggerResultRendering), template result/result_panel.html (Results/Sources sub-tabs, hyperscript toggle; Sources hx-trigger="click once") → result/result_graph.html. Charts: the result_charts esbuild bundle (theme/static/scripts/result_charts/{index,chart,config,legend,tooltip,display}.js → bundles/result_charts.js), data = window.emissions (ModelWeb.system_emissions). Split legend = legend.js splitCapsuleLegendPlugin; panel slide = hammer_utils.js (displayPanelResult / hidePanelResult).
Sankey (Phase C)
Routes sankey-cards/, sankey-form/, sankey-diagram/, sankey-delete-card/ in adapters/views/sankey_views.py (ANALYSE_BY_CHIPS, DEFAULT_ACTIVE_COLUMNS, EXCLUDABLE_CLASSES, _build_sankey_payload → library ImpactRepartitionSankey). Templates result/sankey_section.html / sankey_card.html / sankey_diagram.html; rendered by ECharts in theme/static/scripts/sankey.js (reads data-sankey; sankeyToggleChip). Settings persist in repository.interface_config["sankey_diagrams"].
Sources & export (Phase D)
Route source-table/ → source_table (template result/source_table.html, rows source_table_row.html; inline editor source-table-row-editor/<id>/<attr>/ → source_table_row_editor.html, metadata-only via source_metadata.js). Export download-sources/ → download_sources (openpyxl, sheet "Sources"; button label literally "Export to xslx").
Derivation surfaces
Inline formula: side_panels/edit/calculated_attributes_section.html + calculated_attributes/components/ancestors_formula_and_children.html via get_calculated_attribute_explanation/<id>/<attr>/ (hx-trigger="click once"). Hourly attrs add a timeseries chart (get_explainable_hourly_quantity_chart_and_explanation/…, calculated_attributes_timeseries.js). Calculus graph: display-calculus-graph/<id>/<attr>/ builds the pyvis graph (library build_calculus_graph), caches it (SessionCacheRepository namespace calculus_graph), serves via graph/<cache_key> in an iframe, opened target="_blank".
Not in the panel (clarifications)
No hourly chart in the Results panel — only yearly/monthly bar + cumulative line + Sankey; the only hourly chart is the per-attribute explain chart in the edit panel. No centered explain modal. The model exports as JSON (not xlsx); xlsx is sources-only.
