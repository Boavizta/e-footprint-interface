save-and-load

There are no accounts and no server-side library here — a model lives in your session, and persisting it means moving a file. save-and-load is that round-trip: Export downloads a .e-f.json, Open file reads one back, and a single model or the whole two-model workspace travels the same way. Success isn't "a file downloaded" — it's a faithful round-trip: the model (and its Sankey/UI settings) comes back intact, recomputed and verified, and when a file can't be opened the app never loses your work — it routes you to a recovery page that still hands it back.

Faithful to the deployed app. Draws from the per-model toolbar (upload_download_reboot_model_tooltips.html), the Open a file side panel, the download-json / download-workspace / upload-json views, and the standalone recovery.html. Code-level pointers — the file documents, content-routing, session persistence — are collected for agents in Implementation pointers at the foot.

1

Save = download a file

"Save" is a download. A single-model session exports straight to a .e-f.json; a two-model session lets you take just the active model or the whole workspace in one file. The export deliberately drops calculated attributes (they're recomputed on load) and carries your UI settings along — so the file stays small but round-trips faithfully.

Two modelsExport ▾ → This model (download-json) or Both models (download-workspace, one envelope).

2

Load = open a file

One Open file entry handles everything. You pick a .e-f.json and hit Open; the app inspects the contents and routes itself — a single-model file replaces the active model, a workspace file restores both slots. Either way the model is re-deserialized and its calculated attributes recomputed, so what you see is verified, not trusted from the file.

One entry, content-routedThe same Open handles single vs workspace files — decided by the file's contents, not its name.
Recomputed on loadCalculated attributes are rebuilt on import; the model is migrated forward if it's from an older schema.

Because the file is validated and the model rebuilt before it can take over, a bad upload can never leave you on a broken canvas. The error names what went wrong — wrong format, malformed JSON, or a valid file that isn't an e-footprint model — and the only move is to back out; your current model stays exactly as it was.

3

Recovery — get your work back, don't lose it

If the session model can't be loaded — a corrupt file, an incompatible version — the app doesn't crash to a 500. It shows a recovery page that is fully decoupled from the model (it has to render when the model is exactly what's broken). The raw stored data is still there, so you can download it verbatim before resetting, and report the bug — with versions and the error type only, never your model data or a traceback.

Decoupled by designThe page reads only the slot index and a cache check — it never deserializes, so it works on a dead model.
Raw downloadPer slot — Download your current model (one) or Download model A / B (two) — serves the stored JSON verbatim.
Reset to blankA green Reset to a blank model (POST, confirmed) discards to an empty model and re-opens the picker.
Safe bug reportThe GitHub link prefills versions + error type only, and asks you to attach the file you just downloaded — no data leaks into a public issue.
Always reachableAlso at /recover/ and as the production 500 handler — a read-only escape hatch that's always available.

What's in the file

One extension, .e-f.json, for both shapes — the app routes by content, not extension, so a single-model session can still open a workspace file. Models export without calculated attributes (recomputed on load) but with their UI settings (Sankey views, etc.), so an analysis round-trips intact while the file stays small.

A model is held only in your session (a cache with a short TTL) — there's no account and no server-side saved-models library. Closing the tab and coming back later is exactly what export/import is for.

Decisions taken

DecisionWhy / why-not-X
One .e-f.json extension for both single-model and workspace files; route by content, not extension.A single-model session can still open a workspace file and vice-versa — the user never has to know which kind they have. The workspace file is an additive envelope of byte-for-byte single-model documents, so the single-model format is never re-implemented or allowed to drift.
Calculated attributes are stripped on export and recomputed on import.Keeps files small and makes import authoritative — what you see is recomputed from inputs, not trusted from a possibly-stale or hand-edited file. UI-only state (interface_config) rides inside the file but the model never touches it; the session repository owns it.
Loading is POST-only; the recovery download and /recover/ are GET.A bare GET must not mutate the session model. But a read-only escape hatch has to stay reachable even when the model is what crashes the deserializer — so the recovery reads and the raw download are safe GETs that never build the model.
The recovery page is fully decoupled from the model.It must render precisely when the model is broken, so it reads only the slot index + a cache-existence check and never deserializes. Anything that touched the model could fail the same way the builder just did.
The bug report carries versions and the error type only — no traceback, no model data.The report opens a public GitHub issue; leaking model contents or a stack trace there would be a privacy and noise problem. The user is asked to attach the file they just downloaded if they choose to.
Destructive confirms are decided at click time from the live DOM, not baked into the markup.The toolbar outlives partial swaps, so a baked-in hx-confirm would go stale; reset only prompts when the model is actually non-empty, read live at click time.
"Open file" (replace here) and "+Add → Import" (add as second) are distinct intents with separate panels.Opening-into-the-current-slot and adding-a-second-model-to-compare are different actions; collapsing them into one would make it ambiguous whether a file replaces or augments your work.
⌗ Implementation pointers for agents — the file documents, content-routing, recovery & session persistence

Where this lives in code. Start with specs/architecture.md ("Persistence and interface_config", the workspace / multi-slot session). The toolbar is upload_download_reboot_model_tooltips.html (Open file · Export / Export ▾ · Reset).

Export (Phase A)
download-json/ → download_json (_single_model_document = model_web.to_json(save_calculated_attributes=False) + interface_config + version; filename {date} UTC {name}.e-f.json via sanitize_filename/smart_truncate); download-workspace/ → download_workspace (envelope {efootprint_workspace_version, active_slot, models:[…]}). Toolbar single <a> vs two-model Export ▾ dropdown (#download-menu-toggle).
Open / upload (Phase B)
open-import-json-panel/ → open_import_json_panel → side_panels/import_model.html (header_name="Open a file", save_button_label="Open", label Choose a file (.e-f.json), native <input type="file">, no drag-drop). upload-json/ → upload_json: content-routes on "models" in data → views_workspace._restore_workspace, else single-model (upgrade_system_data → ProgressiveImportService.import_system → distinctify_against_siblings → persist_to_cache) → redirect("model-builder"). Failure → re-render with error-import-modal (modals/import_error_modal.html); messages in upload_json.
Recovery & reset (Phase C)
render_recovery_page (from model_builder_main's try/except), recover/ → recover_model, project handler500 → server_error; template model_builder/recovery.html (never references model_web). Raw download download-raw-json/?slot=N → download_raw_json (serves get_system_data() verbatim). Bug URL build_report_bug_url (versions + error type only). Reset reset-model/ → reset_model (POST; scratch baseline; re-opens picker). Click-time confirms in model_builder_main.js (data-confirm-when-model-not-empty).
Persistence
adapters/repositories/session_system_repository.py (cache key system_data:{session}:{slot}; Redis TTL 600s / Postgres 43200s; save_data merges interface_config + version; MAX_PAYLOAD_SIZE_MB ≈ 30, a shared budget across both slots → PayloadSizeLimitExceeded on save), session_workspace_repository.py, workspace_index.py; serialization/migration in progressive_import_service.py, version_upgrade_handlers.py; library system_to_json, duplicate_system.
Not built (clarifications)
No drag-and-drop and no client-side validation — a plain native file input, validated server-side only. No accounts / named server-side saved models — session cache only (a model is lost when its TTL lapses unless exported). The Sources xlsx is the only spreadsheet; the model itself is JSON.
