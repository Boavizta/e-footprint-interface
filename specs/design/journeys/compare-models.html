compare-models

You've built a model; now you want to know whether a change actually helps. compare-models sets two computable models side by side and quantifies the difference — the common path is duplicate the current one, tweak the copy, read the delta. Success isn't "two models open" — it's a clear, honest comparison: how much CO₂e separates them, what explains the gap, and which inputs differ. Comparison is additive and non-destructive — both models stay exactly as you left them; the dashboard is a fresh projection you open and dismiss without touching either.

Faithful to the deployed app. Draws from the model tab strip, the ＋ Add a model to compare dropdown, the compare/ dashboard (KPI strip · decomposition · paired & cumulative charts · assumptions diff) and the comparison service. Code-level pointers — routes, the resident-sibling reveal, the comparison service — are collected for agents in Implementation pointers at the foot.

Two models, one workspace

Compare — the dashboard, in place

⇄ Compare doesn't navigate; it reveals a resident dashboard over the hidden builder, rebuilt fresh from both models each time. One identity-colour language runs throughout — Reference blue, Comparison amber; reductions green, increases red. Every figure is a full-timespan sum, and the delta convention is fixed: second model minus first.

Reveal, not navigateThe dashboard is a resident region toggled into view — so dismissing it is instant (Phase D). It isn't a URL.

5

Dismiss — back to building, nothing lost

Dismissing is the reveal in reverse: the builder was never torn down, just hidden, so an in-progress edit resumes intact and the return is instant. Clicking the same model you came from warns about nothing; the only place an unsaved-edit guard can fire is switching to the other model on the way out — see the mechanism below.

Instant returnA client-side reveal — no reload; the dashboard fragment is emptied behind it.
Nothing lostA side panel or help drawer open before Compare rode along hidden and resumes intact — opening Compare discards nothing.
Cross-model guardLeaving to the other model can trip the unsaved-changes guard — landing over the now-revealed edited model (mechanism fold).

#comparison-view — revealed sibling, the dashboard
Resident siblingThe dashboard is a sibling of the builder, not a replacement. Opening Compare only adds d-none to the builder; dismissing removes it. Both canvases — and any open panel — stay in the DOM the whole time.

The old design swapped the dashboard into the main content block, destroying both canvases and forcing a ~1.2 s full reload on return. Making #comparison-view a resident sibling toggled by visibility turns dismiss into an instant client-side reveal — and makes opening Compare non-destructive: an open side panel rides along hidden.

Open = no warningOpening Compare discards nothing, so it deliberately does not trip the unsaved-changes guard.
Dismiss, same modelA capture-phase handler tears the dashboard down first, then cancels the navigation — the preserved panel just resumes. No warning.
Dismiss, other modelThe switch proceeds; because switching is a discarding path, the guard — "The currently opened side panel has pending changes. Would you like to continue and lose them ?" (No, Cancel / Yes, Continue) — lands over the revealed edited model.
Remove-model guardRemoving a model has its own confirm, made unsaved-aware — it appends "You also have unsaved changes in the open panel that will be lost."

Decisions taken

DecisionWhy / why-not-X
Comparison is a dedicated dashboard tab — additive, not a split-screen or a results-panel mode.The per-model workspaces stay exactly as they are; the dashboard is a third surface laid on top. A split-screen or a "compare mode" inside Results would entangle two models in one workspace and complicate every builder interaction.
⇄ Compare is disabled until two complete models exist — never offered then errored.Comparing reads each model's footprint; an incomplete model has none. The tab gates on the same readiness as Results, and names the unblock step — the disabled-instead-of-error rule, applied to comparison.
The dashboard is a resident in-flow sibling, not a content swap or a floating overlay.The old swap destroyed both canvases and cost a ~1.2 s reload on return. A floating overlay would fight the body-level leader-line SVGs and force a brittle z-index between the chrome bars. A resident sibling toggled by visibility makes dismiss instant and Compare non-destructive.
One tab strip — the dashboard owns none.Two strips would mean duplicate element ids, breaking the id-based switch JS. The single strip stays visible while comparing (⇄ Compare highlighted) and is how you dismiss.
Compare is not a URL.It's an ephemeral projection of two models, rebuilt fresh — not a navigable resource. And a refreshed /compare/ would error once a model's session slot expires, so it deliberately doesn't push to the address bar.
Tabs are fixed role labels (Reference / Comparison), not model names.A long name can't stretch a fixed-width tab; the editable name lives in the toolbar. Roles also map cleanly to the Δ convention (second minus first).
Figures are full-timespan sums (v1); decomposition merges Servers + Storage but keeps the full taxonomy.Full-timespan sums are exact and need no normalization — the honest limitation is stated per card. Merging Servers + Storage by label keeps the bars readable while still summing exactly to the headline Δ.
The inputs diff matches objects by id first, then name + type.The common workflow is duplicate-then-edit, so id-matching means a rename isn't reported as a change; falling back to name + type still diffs two independently-built models. Identical inputs are omitted to keep the diff to what matters.
