onboarding First run

The first ninety seconds. onboarding takes a newcomer from the home page to a canvas they can actually start on — pick a starting point, land on a model (pre-filled or blank), and get oriented by a guided tour, with help one click away at every step. Success isn't "they reached the builder" — it's knowing where to begin: the tour says start in the middle, with a usage journey, the canvas steers them there by disabling what isn't ready yet, and every concept has a help drawer behind a ?.

Faithful to the deployed app. Draws from home.html, the first-run template picker (onboarding/template_picker.html), the canvas, the driver.js guided tour (tour_steps.py + tour.js) and the help drawer. Note: "how-to guides" are not a separate picker category — they attach to template cards. Code-level pointers are collected for agents in Implementation pointers at the foot.

Pick a starting point

The picker appears only when the model is empty — a returning user with work in progress goes straight to their canvas. A template pre-fills a realistic model to learn from; Start from scratch opens an empty one; Load a json file opens the import panel. Whatever they choose re-renders the whole builder; closing with × just reveals the empty canvas behind.

Shown when emptyDriven by a server-side emptiness check, not a "seen" flag — so it can't be mistaken for real content, and never nags a returning user.
Two groupsTemplates (e-commerce, AI chatbot, Industrial IoT, ML workflow) and Or (scratch / upload). How-to guides aren't a third group — they ride on template cards as "… ↗" links.

Land on the canvas — and know where to begin

The newcomer arrives at the same three-column canvas the rest of the app uses (the build-a-model loop). It's either pre-filled from a template — a worked example to poke at — or the empty My system. Either way the disabled, never errored rule does the teaching: the one always-available action is Add usage journey, the recommended place to start.

Empty or pre-filledScratch → empty My system; a template → a realistic model already on the canvas.

4

The guided tour — once, and replayable

The tour orients the newcomer on a real canvas (it waits for the picker to clear). It walks the columns in modeling order — journeys → infrastructure → usage patterns → results — then points out edge modeling, comparison, and where help lives. It runs once ever and never nags again; the Help menu replays it on demand.

Once, everAuto-runs a single time (remembered locally), deferred until the picker is gone so it never lands on the bare picker.
The pathUsage journeys → infrastructure → edge toggle → usage patterns → Results → Compare → the help "?" → the Help menu.
Blank-model stepOn a scratch start it appends a first-action step: "Suggested first step: create a usage journey."
Non-blockingOverlay-click closes it; the toolbar and an open help drawer stay above the dim, so the user can explore mid-tour.
MobileOn phones the help step doesn't open the drawer (shows alternate copy); chrome anchors fall back to the menu button.

Model what a user actually does with your service — a sequence of steps, each calling jobs on your infrastructure.

Common pitfalls

Don't confuse a journey (what users do) with a usage pattern (how many users, and when).

Interactions

Steps run jobs on servers and edge functions; usage patterns point at a journey to drive it.

5

Help, always one click away

Depth is never more than a click off the canvas. Each object type has a ? beside its Add button that opens a help drawer explaining the concept — when to use it, what to watch for, how it connects — plus field tooltips and info icons in the forms. The Help menu collects the meta-actions: replay the tour, re-open the templates, open the docs.

? on every addOpens About {type} — When to use · Common pitfalls · Interactions · a docs link. The same content the tour and field tooltips draw on.
Independent overlayThe drawer is its own panel; opening an edit side panel or switching model closes it, so the two never fight.
Help menu↺ Replay tour · ▦ Open templates · 📖 Documentation — the home for re-orienting any time.

Decisions taken

DecisionWhy / why-not-X
The picker is shown by a server-side emptiness check, not a localStorage "seen" flag.A returning user who never actually built anything still gets guided, and a metadata-only key can never be mistaken for real content. Tie the picker to the thing that matters — is there a model? — not to whether a browser has visited before.
The picker is an overlay rendered into the canvas, not a separate page or a Bootstrap modal.Picking re-renders the whole builder in place, so the model and chrome stay intact; closing just reveals the canvas. A separate page would add a navigation and a reload to the very first step.
How-to guides attach to template cards as links — they're not a third picker category.A guide isn't a starting model; several guides can describe one scenario. Keeping them as per-card "… ↗" links avoids duplicating cards and keeps the picker to two clean groups.
The tour auto-runs once, deferred until the picker is gone, and is non-blocking.It must orient on a real canvas (never the bare picker), must never nag on return, and must not trap the user — overlay-click closes it and help stays reachable for its whole duration.
Tour copy lives server-side, resolved through the one description provider.Concept prose has a single source of truth shared with the help drawer and tooltips; nothing about a domain concept is duplicated in JavaScript to drift out of date.
"Add usage journey" is never gated; "Add usage pattern" is gated until a journey exists.The canvas itself teaches the recommended starting point — the tour's "start in the middle" — through disabled-instead-of-error, instead of relying on the user to read instructions.
Loading a model is POST-only; the docs deep link (/template/<id>/) is the one GET exception.A bare GET must not mutate the session model — except a documentation link click is itself the user's intent and loading is idempotent, so the deep link is allowed to load and redirect.
