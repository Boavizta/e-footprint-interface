build-a-model

The core loop — a product person assembling a model on the canvas. They add the pieces of a digital service (a usage journey, the servers behind it, a usage pattern that drives it), then tune each one's parameters until it matches reality. Success isn't "an object created" — it's a complete, computable model: enough connected objects that the Results button lights up. The canvas never lets the user reach a dead end — actions that aren't possible yet are disabled with a reason, not offered then rejected — and every change patches the canvas in place, no full reload. The same loop builds both web and edge models; the Edge toggle (Phase E) just reveals more object types.

A functional walkthrough, faithful to the deployed app. Code-level pointers — files, routes, the in-place-update mechanism — are collected for agents in Implementation pointers at the foot of the page.

1

The workspace — and what you can't do yet

The canvas is the home surface. The user reads it left-to-right but builds it right-to-left: infrastructure and journeys first, then the usage pattern that ties them together. Rather than offer every action and reject the impossible ones, the app pre-disables what isn't ready. Add usage pattern stays greyed until a usage journey exists; hovering it shows exactly the unblock step. This is the app's defining UX rule — disabled, never errored.

Gated addA not-yet-possible Add button is greyed; hovering it names the unblock step — e.g. "Add a usage journey in the Usage journeys section first." The reason is specific to what's missing.
Results lockedThe bottom Results bar is dimmed with "Complete your model to access results" until the model is computable.
Edge sections hiddenThe Edge toggle is off, so the edge Add buttons and lists aren't shown. Turning it on reveals them — see Phase E.
Help on demandThe ? beside each Add button opens a help drawer with the object type's description and field tips — depth without leaving the canvas.

Add — fill the form, Save

The panel is the single surface for both add and edit. On add it carries only the fields needed to create a valid object — advanced ones are collapsed by default behind "Show advanced parameters," so a newcomer isn't faced with the full parameter set. The footer note sets the expectation: depth comes later, on the card. Save creates the object and closes the panel.

Quantity fieldA value + a unit dropdown, with an optional source / confidence (High · Medium · Low · Not set) — every number can carry its provenance.
Advanced collapsedAdvanced fields start hidden; the toggle flips to "Hide advanced parameters." A field that fails validation auto-expands its section, so an error is never hidden.
Unsaved-changes guardClosing the panel with pending edits prompts "The currently opened side panel has pending changes. Would you like to continue and lose them?"

3

Saved — the canvas patches itself in place

A change never reloads the page or re-renders the whole canvas — only what changed is patched: the new card drops into its column, and if a prerequisite was just satisfied the affected Add buttons unlock, with a toast pointing them out. The user watches the model gain a piece and gain a capability in one beat — the disabled-instead-of-error rule running in reverse.

New card in placeThe new card appears in its column; the rest of the canvas is untouched — no scroll jump, no flicker.
Constraint unlockSatisfying a prerequisite re-enables the Add buttons it gated and fires a "You can now add …" toast. The reverse of the gate in screen 1.
Toward ResultsEach add moves the model toward computable. Once a usage pattern connects a journey to its infrastructure, the Results bar lights up (its own journey — view-results).
Edge object addedAdding the first / removing the last edge object also flips the Edge toggle's state (and locks it while edge objects exist). See Phase E.

4

Edit — the panel reopens with the full object

Add is deliberately shallow; edit is where depth lives. The same drawer comes back with every attribute (including the advanced ones), the object's links to other objects, and a read-only Calculated attributes breakdown — its derived values with their full derivation, there to audit what the model currently computes. Save closes the panel and patches the canvas in place — so the breakdown is an audit of the saved state, not a live preview of an unsaved edit.

Object linksAn object's editable relationships are managed right here — a list, or a weighted dict — all using one shared widget family (see Relationships). Some single links, like which server a job runs on, are fixed when the object is created and don't reappear as editable fields.
Reverse membershipIf this object is held in some parent's weighted dict (a step in a journey, a job in a step, a device in a group), the panel also shows a membership section — which parents hold it, at what count, with an × to remove. A plain server isn't, so it shows none. (See Relationships.)
Calculated attributesA read-only breakdown for auditing the object's derived values, each with its full derivation — the formula and calculus graph behind every number, its own methodology journey: audit-a-result. It reflects the currently saved state — not a preview of unsaved edits, since Save closes the panel.
Edit patches in placeSave updates only the edited card and anything its change affected (a constraint, the edge toggle) — never a full reload.
Rename modelThe model's own name is edited from a separate "Rename your model" panel, reached from the tab — not from an object card.

5

Delete — confirm, then patch out

Deletion is a two-step the moment it has any consequence. The plain case asks a simple yes/no; Yes, delete removes the card, re-locks anything that depended on it, and closes the panel. But before the confirm even appears, the app checks whether the object is safe to remove — and if it isn't, it doesn't offer a destructive button at all.

Plain confirm"Are you sure you want to delete this {type} ?" → Yes, delete (red) / No, go back.
Cascade warningIf the object owns children, the message becomes "This {type} is associated with N {child}. This action will delete them all," with a note that shared children used elsewhere stay. (See fold.)
Blocked, explainedIf another object depends on it, there's no delete button — a "can't delete" modal names the blockers and the unblock step instead. (See fold.) The disabled-instead-of-error rule, applied to deletion.
Re-lock on deleteRemoving the last prerequisite re-locks the buttons it had enabled, with a "… is no longer available" toast — symmetric with screen 3.
Blocked Can't delete — referenced by another object — the dependency is named, no destructive button shown

E

Edge modeling — the same loop, more object types

Edge isn't a separate way of working — it's the same canvas, the same side panel, the same in-place patches and the same gating, with extra object types switched on. The Edge toggle reveals the edge Add buttons and lists; turning it off again hides them (only once no edge object remains). The one genuinely new shape edge introduces is the edge device group — a hierarchical fleet — covered in Relationships rather than duplicated here.

Toggle revealsOn shows the edge Add buttons and lists alongside the web ones — the same web/edge fork the whole canvas keys off.
Toggle locks onWhile any edge object exists the toggle is locked on, with "Remove edge objects to turn off" — the disabled-instead-of-error rule guarding the paradigm itself.
Edge object typesEdge usage pattern / journey, edge device, edge device group, and the recurrent needs — each gated the same way (e.g. "Add an edge device in the Infrastructure section first.").
Everything else identicalAdd / edit / delete, the side panel, in-place patches, the Results gate — all the same as Phases A–D. Edge changes the object palette, not the loop.

Relationships — one mechanic, several shapes

Modeling is mostly connecting objects, and the app gives every connection the same UX: a vertical list of what's attached (each row with an × to remove, plus ↑↓ to reorder where order is meaningful), then a reuse select + ＋ to attach an existing one. Learn it once, it works everywhere — the row widget is literally shared; the shapes differ only in what each row carries. Objects are linked, not duplicated: a parent attaches objects that already exist, so each stays a single source of truth.

From the other end — managing the relationship from the child's panel

A weighted dict is editable from both ends. The parent's panel edits the forward dict (a journey's Steps, above); the child's panel edits the same relationship in reverse — a step's panel shows which journeys use it, at what count, and lets it join another. Both write the same data, so the two views never disagree.

The reverse view is the "Reverse membership" state from screen 4, made concrete. A step doesn't own which journeys use it — yet from its own panel it can adjust the count it carries in each, detach from one, or attach to a new one, all writing the same relationship. The section updates in place after each change.

Same count, both endsEditing the count here is the same weighted value the parent's widget shows — one source of truth, two editors.
Add to another parentA select of parents this child isn't in yet + a labeled "Add to journey" button (its wording matches the relationship — clearer than a bare ＋ in the reverse context).
Dict children onlyOnly objects held in a weighted dict get a reverse section — there's a count to manage. A list-link child (an edge function in a journey) has no per-membership weight, so it's managed only from the parent's list.

On the canvas — edit the weight in place, no panel

The weight of a dict membership isn't only editable in a panel. Wherever a weighted-dict child is shown nested on the canvas — a step under its journey, a job under its step, a device or sub-group inside an edge device group — its count is editable right there as an always-visible × n field. Changing it writes the same count the panel widgets do and recomputes in place — no panel, no reload. So the same weight has a third editor alongside the two panels above; all three stay in sync.

Decisions taken

DecisionWhy / why-not-X
Actions that aren't possible yet are disabled with a tooltip reason, not offered then rejected.The primary user is non-technical, so a dead end is worse than a missing button. The tooltip names the exact unblock step, turning a block into guidance.
Every change patches the canvas in place, never a full reload.The canvas can be large; re-rendering all of it on every small change is wasteful and loses scroll & focus. Only the regions that actually changed are patched — the model stays live without a reload.
One right-docked side panel is the single surface for both add and edit.Add carries only what's needed to create a valid object (advanced fields collapsed); edit reopens the same drawer with the full object — attributes, links, any reverse membership, calculated outputs. Light create path for newcomers, all depth one click away on the card.
Add forms collapse advanced parameters by default; "edit later on the card" is stated in the form.The full parameter set overwhelms the primary user. Defaults get them a valid object; the footer note sets the expectation. A validation error auto-expands its section, so nothing breaks silently.
Delete is confirm-or-explain: a consequence-free delete asks yes/no; a blocking one explains why and offers no destructive button.The model must never be left broken, and the user must understand scope. A safe delete confirms; an owning delete spells out the child count; a blocked delete refuses and names the fix.
Every relationship shares one UX; they differ only in what each row carries.Why-not-bespoke-widgets: one row-list / reuse-select-＋ / × pattern for list links, weighted dicts and nested edge groups (a single link is its 1:1 cousin) means the user learns connecting once. Objects are linked, not duplicated. Full treatment in Relationships.
Web and edge are one journey behind a toggle, not two — edge adds object types, not a new loop.The add/edit/delete loop, side panel, in-place patches and the gate are identical; only the object palette and one new relationship shape (the nested edge device group) differ. Splitting them would duplicate all of that and let the two drift. The toggle locks on while edge objects exist, so they can't be orphaned.
