audit-a-result Methodology

A footprint is only useful if you can defend it. audit-a-result is the methodology view: every number the model produces can be traced — down through the formula that computed it to the cited, confidence-rated inputs it rests on. Success isn't "a result appeared" — it's being able to answer "where does this number come from, and how sure are we?" for any value on screen. The two halves are provenance (every input has a source & confidence) and derivation (every output explains itself), meeting in one unbroken trail.

Faithful to the deployed app. Draws from the quantity fields' source / confidence, the Sources tab (source-table), the edit panel's Calculated attributes formula expander, and the calculus graph. Derivations are tooltips, inline expanders and a new-tab graph — not modals. The surfaces are reached from view-results (Sources) and build-a-model (the edit panel); this journey is their methodology home. Code-level pointers for agents in Implementation pointers at the foot.

Why its own journey. Today these surfaces are distributed — provenance in the form fields and the Sources tab, derivation in the edit panel — and easy to miss. As auditability grows into a first-class feature, this is where the consolidated story belongs; the other journeys point here rather than re-explaining it.

Every input carries a source and a confidence

Auditability starts at the input. Each quantity a modeller enters can be tagged with where it came from and how sure they are — captured on the field itself while editing. It's metadata about the value, not a modeling input: it describes how well the number is backed, without changing the number.

Per-value provenanceA source (name + optional link) and a confidence attach to each input quantity.
Confidence levelsA qualitative flag — Confident · Uncertain · Not provided — not a false-precision number.
Metadata, not inputEditing source / confidence is a metadata-only change — it never recomputes the model.

The Sources tab — provenance, consolidated

Scattered field-level provenance is gathered into one place: the Sources tab of the Results panel lists every input behind the model with its source and confidence, all editable in place. It's the auditor's-eye view — scan what's solid, what's Uncertain, and what still has no source — and export it for a review.

Every calculated value explains itself

A computed number isn't a black box. In an object's Calculated attributes, each derived value can be unfolded into the exact formula that produced it, the input variables it draws on, and what it in turn feeds. The symbols in the formula are tooltipped, so the equation reads on its own.

The full trace — the calculus graph

The inline formula shows one step; the calculus graph shows them all. Opened in a new tab, it's the complete tree from the chosen value back through every operation to the raw inputs — the artefact you'd hand a reviewer who asks to see the whole computation, not just the headline.

Whole treeEvery ancestor input and operation down to the result — the full derivation, not one step.
New tabOpens out of line so it never crowds the panel; interactive — drag to explore.
Two ways inThe formula's See the calculus graph link, and the Sources tab's (see graph) on calculated rows.
CachedBuilt once and cached server-side, then served by key — the heavy artefact never bloats a normal response.

Decisions taken

DecisionWhy / why-not-X
Source & confidence are metadata on a value; editing them never recomputes the model.Provenance describes how well a number is backed, not the number itself — so a metadata-only edit (fixing a source, downgrading a confidence) is fast and side-effect-free, and the table doesn't reload.
Confidence is a qualitative flag — Confident / Uncertain / Not provided — not a percentage.A made-up uncertainty number would imply rigour the input doesn't have. A three-level flag is honest, quick to set, and enough to triage what needs a better source.
Calculated values aren't sourced — they link to their derivation instead.A derived number's "source" is its formula and the inputs beneath it. Asking for a citation on a computed value is a category error, so its row offers (see graph) rather than a source field.
Modeling truth — the formulas and the calculus graph — is the library's; the interface only renders it.Derivations come straight from the engine that computed them, so they can't drift from what actually ran. The browser lays out and displays; it never re-derives.
Derivations escalate through lightweight surfaces — tooltip → inline formula → new-tab graph — never a modal.A glance wants a tooltip; the formula wants to stay beside the value; the full tree is large, so it's cached and opened out of line. A heavyweight modal would interrupt the read and bloat every response.
⌗ Implementation pointers for agents — the provenance fields, the Sources table & the derivation / calculus-graph views

Where this lives in code. Start with specs/architecture.md (the render layer; metadata-only edits). The surfaces are reached from view-results (Sources) and build-a-model (the edit panel's Calculated attributes).

Provenance (Phase A)
Source / confidence are fields on a quantity (rendered in the dynamic form fields). Consolidated view: route source-table/ → source_table → result/source_table.html + source_table_row.html; inline editor source-table-row-editor/<id>/<attr>/ → source_table_row_editor.html, posting via edit-object but metadata-only (no table reload — source_metadata.js). Export download-sources/ → download_sources (openpyxl, sheet "Sources"; button label literally "Export to xslx"). Labels via adapters/label_resolver.py.
Derivation (Phase B)
side_panels/edit/calculated_attributes_section.html (shown when object_belongs_to_computable_system) + calculated_attributes/components/ancestors_formula_and_children.html (Input Variables · Formula · Direct children) via get_calculated_attribute_explanation/<id>/<attr>/ (hx-trigger="click once"). Hourly: get_explainable_hourly_quantity_chart_and_explanation/… → calculated_attribute_chart.html (calculated_attributes_timeseries.js). EcoLogits: eco_logits_calculated_attribute_explanation.html.
The calculus graph (Phase C)
display-calculus-graph/<id>/<attr>/ builds the pyvis graph (library efootprint.utils.calculus_graph.build_calculus_graph), caches the HTML (SessionCacheRepository namespace calculus_graph), serves it via graph/<cache_key> in an iframe (calculus_graph.html); links open target="_blank".
Tooltips
Chart tooltips in the result_charts bundle (tooltip.js / display.js); Sankey node/link tooltips precomputed server-side in adapters/views/sankey_views.py and rendered by sankey.js.
Forward-looking
These surfaces are currently distributed across the Sources tab, the form fields and the edit panel. If auditability becomes a first-class destination (e.g. a dedicated audit view or a per-result "explain" entry point), this journey is the canonical description to evolve.
