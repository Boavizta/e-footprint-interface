# Source metadata — implementation plan

**Status:** Plan — under review.
**Date:** 2026-04-28.
**Spec:** [`spec.md`](spec.md).

## 1. Approach

Two attributes — `confidence` and `comment` — are added to `ExplainableObject` (library), default `None`. They are user-settable on inputs only; they stay `None` on calculated values. Both fields plus `source` are loaded centrally by `json_to_system` after the per-subclass `from_json_dict` returns the constructed instance — subclass overrides don't need to know about any of the three. Serialization is already centralized in the base `to_json`, which we extend to emit `confidence` and `comment` (omitted when `None`) and the new `"source": "<source_id>"` ref. As payload fields on each ExplainableObject the two new attributes are purely additive (absent fields read as `None`).

Source identity is fixed at the JSON-shape level by hoisting `Source` to a top-level `"Sources": {id: {...}}` block in the system JSON, mirroring how `ModelingObject` instances are already laid out. Each `ExplainableObject`'s `to_json` emits `"source": "<source_id>"` (a string ref) instead of an inline `{"name": ..., "link": ...}` blob. `Source` gains a stored `id` generated at `__init__` using the same pattern as `ModelingObject` (`css_escape(name) if Source._use_name_as_id else str(uuid.uuid4())[:6]`); the `_use_name_as_id` flag is flipped in test contexts via `conftest.py` and stays `False` in production. The id is persisted on the instance and round-trips through JSON, so within one file every `ExplainableObject` referencing the same source by id resolves to the same `Source` instance after load. The two sentinels — `Sources.USER_DATA` and `Sources.HYPOTHESIS` — are constructed with explicit, hardcoded ids (e.g. `id="user_data"`, `id="hypothesis"`) so they re-identify with the live Python singletons across processes, regardless of where the JSON came from; the `json_to_system` pre-pass swaps any source matching a sentinel id for the corresponding canonical instance. The other `Sources.*` constants (ADEME, REN, etc.) are just default-model conveniences — within one loaded model they're shared by id-ref like any other source, and Python-identity with the original `Sources.ADEME_STUDY` constant across reloads is not required. `compute_classes_generation_order` already orders generation by reference dependency; `Source` slots in as a leaf class with no dependencies, generated first. Result: on-disk dedup, structural identity within a file, sentinel preservation across reloads. This is a schema change, not additive — bumps the library to `21.0.0` and ships an `upgrade_version_20_to_21` handler that rewrites old inline source dicts into top-level entries + id refs.

On the interface, the existing `source.html` partial — today a client-side label derived from "is the input value different from default?" — is replaced by a server-rendered, interactive source-and-comment line backed by the live `ExplainableObject.source` / `.comment`. A new confidence-badge partial is rendered inline next to the input. The badge dropdown and the source/comment editor are pure client-side widgets that mutate hidden form inputs (`<prefix>_<attr>__confidence`, `__comment`, `__source_name`, `__source_link`) inside the existing edit form; nothing posts on click. The user submits with the existing Save button, which fires the existing `edit_object` POST — the form parser and object factory grow handlers for the four new fields and ride the same write path as every other attribute. The "carry comment, reset confidence on value change" rule is enforced server-side in the factory's edit path so it survives any client tampering. A new `ModelWeb.available_sources` property returns the deduplicated source list scoped to the current modeling, with `Sources.USER_DATA` and `Sources.HYPOTHESIS` always appended.

## 4. Risks

- **Sentinel id drift.** Renaming the `Sources.USER_DATA` / `Sources.HYPOTHESIS` hardcoded ids would silently break every saved JSON that references them — sentinels would no longer resolve to the live Python singletons on reload. Mitigation: a unit test that pins both sentinel ids; treat them as part of the on-disk schema.
- **Centralized metadata application coverage.** Because `_apply_json_metadata` is now the single point that sets source/confidence/comment on loaded objects, missing one of the load paths (main per-class loop, `ExplainableObjectDict` post-pass, lazy `explain_nested_tuples` rehydration) silently drops metadata. Mitigation: a single library-side test that round-trips one instance of every concrete `ExplainableObject` subclass with all three fields set — asserts equality and source identity — and a second test exercising the explain_nested_tuples lazy path with a non-trivial formula.
- **Upgrade-handler correctness.** The handler walks every `ExplainableObject` payload nested under every modeling class. Easy to miss a case (e.g. `explain_nested_tuples` carries inline source dicts too). Mitigation: a frozen v20 system fixture spanning all `ExplainableObject` subclasses + an end-to-end test that loads it via `json_to_system` and asserts the in-memory result matches a freshly-built equivalent.
- **Sentinel emit rule.** Should `Sources.USER_DATA` / `Sources.HYPOTHESIS` always appear in the top-level `"Sources"` block, or only when referenced? Recommendation: emit only when referenced; `from_json_dict` re-identifies them by id against the in-Python `Sources.*` table either way. Avoids polluting JSONs that don't use them.
- **Source-line truth swap.** `updateSource()` in `dynamic_forms.js` currently rewrites the source div on every keystroke, baking in the assumption "any change → user data". With the source line now server-rendered and authoritative, that JS is contradicted. Mitigation: drop the source-rewriting branch from `updateSource`; the server-side factory still defaults the source to `Sources.USER_DATA` on value change (see existing `edit_object_from_parsed_data` line 174), so the actual data follows. Document this in the conventions one-liner.
- **Stale form state across the Save button.** Because all four metadata fields piggyback on the existing Save button, opening the source editor, editing the comment, and then hitting Cancel on the editor must not leak the typed text into the hidden inputs. Mitigation: editor "Apply" writes to hidden inputs; "Cancel" restores the editor's local fields from the hidden inputs and closes. Cover with one E2E case ("type then cancel does not change the saved comment").
- **Comment textarea vs. JSON size.** No data-model cap; recommend a 2 000-char soft cap in the UI textarea (HTML `maxlength`) for layout sanity, as the spec hints.

## 5. Alternatives considered

- **Per-consumer source dedup** (only in the picker, not in the deserializer). Rejected per spec §7: it's symptom-level and leaves the same duplication in the source table and xlsx export.
- **ContextVar dedup at load time** with the on-disk format unchanged (each `ExplainableObject` keeps an inline source dict; `Source.from_json_dict` consults a per-load `(name, link) → Source` registry kept in a `ContextVar` set by `json_to_system`). Cheaper diff, purely additive JSON, no migration. Rejected because it leaves the on-disk duplication intact, makes hand-editing unsafe (change a source name in one ExplainableObject and forget another → silent divergence on reload), and is asymmetric with the `ModelingObject` reference pattern. The structural fix is worth the schema bump given the constitution explicitly waives backwards compat.
- **Confidence/comment on `Source`** instead of `ExplainableObject`. Rejected per spec §2: two values citing the same source can carry different confidence/comments.
- **HTMX `hx-post` per badge click / per editor save.** Rejected: it would create a second write path alongside the existing form-Save flow, double the round trips, and force partial-swap plumbing for state the user is already going to commit through Save. Riding the existing Save button is simpler and consistent with how the rest of the form already works.
- **Dedicated metadata endpoint** instead of reusing `edit_object`. Rejected for the same reason.
- **Client-side enforcement of the "reset confidence on value change" rule.** Rejected as the only enforcement: server-side authority is necessary so the rule survives client-side tampering or stale forms. Client-side animation (`confidence-just-reset` pulse) is kept as UX feedback only.
- **Make `Source` a full `ModelingObject`** (with `modeling_obj_container`, calculated attributes, update functions). Rejected as overkill: `Source` is a pure value-holder. It just needs an id + top-level serialization, not the full ModelingObject machinery.
- **Hash-derived id (deterministic from `(name, link)`).** Rejected: (a) renaming a source would change its id and break every existing reference; (b) the id has to be persisted on the instance anyway (since name can change without the id changing), so deriving it from a mutable field couples two things that should be independent. Storing a uuid-style id at construction is simpler and matches the existing `ModelingObject` pattern exactly.

## 6. Constitutional notes

- **§1.1 (Clean Architecture).** Confidence/comment travel through the existing parser → factory → domain chain. No new layer crossings; templates and views stay in adapters.
- **§1.3 (library is the truth).** All metadata mechanics — including the new top-level `"Sources"` block, the upgrade handler, and source-id determinism — live in `efootprint`. The interface only renders and forwards. Compliant.
- **§1.4 (HTMX-first; minimize JS).** No new client-server round trips: the badge dropdown and the source/comment editor are local widgets that mutate hidden inputs already inside the existing edit form, and the user submits via the existing Save button (which already uses the standard `edit_object` HTMX flow). Vanilla JS is limited to dropdown open/close, editor open/close + apply, the value-changed pulse, and the collision-notice toggle. Compliant.
- **§2.4 (PyPI-pinned `efootprint`).** The library bumps to `21.0.0`. The interface PR pins to `^21.0.0` only after the library is published; until then, `pyproject.toml` and `poetry.lock` reference the local editable path. Standard cross-repo workflow.
- **CLAUDE.md ("no need for backward compatibility, no other external callers").** Justifies the schema change. The upgrade handler exists for the user's own historical JSONs (downloads they may want to re-upload), not for unknown third-party consumers.
- **§4.** Nothing here re-opens a rejected scope item.

No constitutional amendment needed.

## 7. Resolved

- **Sentinel id values.** `id="user_data"` for `Sources.USER_DATA`, `id="hypothesis"` for `Sources.HYPOTHESIS`. Part of the on-disk schema; pinned by a unit test.
- **Sentinel emit rule.** Emit in the top-level `"Sources"` block only when referenced.
- **Empty-string link normalisation.** Empty-string `link` is normalised to `None` in `Source.__init__`.
- **Free-entry collision UX.** Per spec §4 — kept as a distinct `Source` instance, quiet notice in the editor, no auto-snap.
