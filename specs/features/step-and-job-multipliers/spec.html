Step and job multipliers

Status: Spec — implemented · 2026-06-12

1. Problem and audience

  • Problem. A usage journey in e-footprint describes the average interaction a user has with the service. Today every step counts exactly once per journey, and every job counts once per appearance in a step's job list. Two common modeling needs cannot be expressed naturally:
    • "This job fires N times during this step" — the only workaround is adding the same job to the list N times, which is opaque in the UI (the same card repeated), tedious for large N, and impossible for fractional averages (e.g. 2.5 retries on average).
    • "This step happens in only 50% of journeys" — there is no workaround short of manually pre-scaling the step's time spent and creating scaled copies of its jobs, which destroys the legibility of the model.
  • Audience. Everyone who builds usage journeys: primarily the non-technical product person composing an average journey in the interface ("search is used in every visit, checkout in one visit out of ten"), and secondarily technical users driving the efootprint library directly.
  • Why now. Weighting within an average journey is a recurring need in real modeling sessions; the duplication workaround scales badly and the "fraction of journeys" case has no answer at all. A simple per-entry weight closes both gaps without introducing branching-journey complexity.

2. Success criteria

How do we know this feature is done? Each criterion must be testable.

  • A modeler can attach a numeric multiplier (any value ≥ 0, fractions allowed) to each job within a usage journey step — and, identically, to each job within a recurrent server need on the edge side; the modeled number of job executions per journey (resp. per need occurrence) scales accordingly.
  • A modeler can attach a weight (any value ≥ 0, fractions allowed) to each step within a usage journey; the step's contribution to journey duration and all of its job executions scale by that weight (a 0.5-weight step contributes half its time and half its job runs to the average journey).
  • The two multipliers compose: a job with multiplier 3 in a step with weight 0.5 runs 1.5 times per average journey.
  • The multiplier belongs to the relationship, not the object: the same job reused in two steps can carry a different multiplier in each, and the same step reused in two journeys a different weight in each.
  • The default multiplier is 1 everywhere: a model built without touching multipliers computes exactly the same results as today.
  • Existing saved models (JSON exports and in-flight sessions) keep loading without user action and produce unchanged results.
  • Multipliers are explainable: the calculation explanation of journey duration and of job occurrences cites the weight, like any other model parameter.
  • Export → import round-trips multipliers exactly.
  • In the interface, the multiplier is always visible on the canvas next to each step / job entry and is editable from three consistent surfaces: inline on the canvas (with immediate recalculation, no panel round-trip), in the parent's edit panel, and in each child's edit panel — the same experience edge device group counts provide today.
  • A multiplier of 0 keeps the entry in the model but zeroes its contribution; negative or non-numeric input is rejected with a clear validation message.
  • Existing list capabilities are unaffected: steps keep their meaningful order, and adding / removing / reordering steps and jobs works as before.

3. Scope

Capability-level only. File paths, class/method/parameter names, refactor instructions, internal hypotheses, and "mirror pattern X" code-level decisions belong in plan.html, not here.
In scope
  • Per-relationship multiplier on steps-within-journey and on jobs-within-step (web usage journeys), and on jobs-within-recurrent-server-need (the edge→server bridge, which today documents "add the same job multiple times" as its multiplicity idiom).
  • Calculation semantics in the modeling engine: journey duration, job occurrence counts, and every downstream value (device occupancy, impact attribution) consistent with the weights.
  • Persistence: multipliers survive save / load, and older saved models upgrade automatically with all multipliers at 1.
  • Interface editing for both relationships on three consistent surfaces — inline on the canvas (always visible), in the parent's panel, in each child's panel — with validation. Steps/jobs and edge device group counts converge on one shared relationship-editing experience; improvements made along the way benefit the edge surfaces too.
Out of scope (this iteration)
  • Multipliers on other list relationships (e.g. usage patterns → journeys, edge functions → recurrent server needs). Edge-side intensity is already expressed through recurrent hourly volumes and needs no per-entry weight.
  • Branching / conditional journeys (probability trees, alternative paths). Weights approximate variability within a single average journey; modeling distinct journey variants remains the job of separate usage journeys and usage patterns.
  • Per-usage-pattern overrides of a weight (the weight is fixed per relationship, not per pattern).
  • Bulk re-normalization helpers (e.g. "make step weights sum to 1") — weights are free values, not enforced probabilities.

4. UX / interface

The guiding decision: multipliers get one relationship-editing experience, offered on three consistent surfaces — inline on the model-builder canvas, in the parent's side panel, and in each child's side panel. This is exactly the experience edge device group counts already provide; steps and jobs adopt it, and since the machinery becomes shared, any polish lifts the edge experience too.

  • Canvas: an always-visible, directly editable "× n" count sits on every step row of the journey accordion and every job chip of the step accordion — the same inline-count pattern as edge device group entries (change → save → recalculation immediately, no side panel, no Save button). The count is always shown, including when it is 1, so editing it is always one click away and the affordance is discoverable. A count of 0 dims the entry: still in the model, visibly contributing nothing. Unlike edge group entries, step rows and job chips get no inline unlink (✕) on the canvas for now — removal stays in the side panels, to avoid accidental unlinking on the busier journey canvas.
  • Parent side panel: the existing child-list tables gain the numeric column. The journey panel's steps table and the step panel's jobs table — today "selected entries + remove" over a light-gray box, with an "Other existing ones you can reuse" dropdown below — become weighted tables, like the edge group's counts table. Adding an entry starts it at 1.
  • Child side panel: a membership section, mirroring "Group membership". A step's panel lists the journeys that use it, each with its editable "times per journey" and an unlink button, plus an "Add to journey" dropdown; a job's panel does the same for the steps that use it. Same partial, same behavior as edge devices' group membership section.
  • The steps table keeps journey order. Step order is meaningful (it drives when each step happens within the journey), so rows render in journey order — unlike the edge-device counts table, which is unordered and lists alphabetically.
  • Creation panels include the field. When a step is created from the journey card's "+ Add a usage journey step" (or a job from a step's add button), the creation panel knows its parent, so it offers the multiplier field directly, prefilled at 1 — "this job runs 3 times here" is one trip.
  • Wording: "Times per journey" (steps), "Times per step" (jobs in a step), "Times per occurrence" (jobs in a recurrent server need). One mental model — "how many times does this happen here" — that reads naturally for counts (3 API calls) and fractions alike (0.5 ≈ one journey out of two reaches checkout), and avoids the abstract "weight".
  • Validation: the numeric inputs reject negatives (min="0", like the existing counts widget, which snaps invalid input back to the previous value); the model layer enforces the same rule server-side.
  • Results stay explainable in place: the journey's weighted duration appears in its panel's calculated-attributes section, derived from "times per journey × time spent", like every other calculated value.

5. Constraints

  • The library is the domain truth (constitution §1.3): multiplier semantics, validation, and explainability live in the efootprint library; the interface only edits and displays them.
  • Explainability: every computed value carries its derivation; multipliers must appear in those derivations like any other parameter.
  • HTMX-first (constitution §1.4): the editing surface is server-rendered form fields with partial updates, no client-side data layer.
  • Cross-repo delivery: the interface change depends on a released efootprint version; pyproject.toml must reference PyPI before merge (constitution §2.4). Development uses the editable local install per INSTALL.md.
  • Saved-model continuity: the library's JSON format versioning is the upgrade path for pre-feature exports; no user-visible migration step.

6. Open questions

  • None at this stage — the §4 design decisions (wording, three editing surfaces, no canvas unlink for steps/jobs, duration as a calculated attribute, dimmed 0-multiplier entries) are settled.
