Step and job multipliers — Implementation plan

Status: Plan — implemented · 2026-06-12 · Spec: spec.html

1. Approach

Reuse the dict-based-relationship pattern already proven by EdgeDeviceGroup (e-footprint/efootprint/core/hardware/edge/edge_device_group.py): an ExplainableObjectDict[ModelingObject → ExplainableQuantity] whose keys are the related objects and whose values are dimensionless, non-negative quantities. Three plain-list relationships become weighted dicts:

  • UsageJourney.uj_steps: List[UsageJourneyStep] → ExplainableObjectDict[UsageJourneyStep], same attribute name (usage_journey.py).
  • UsageJourneyStep.jobs: List[JobBase] → ExplainableObjectDict[JobBase], same attribute name (usage_journey_step.py).
  • RecurrentServerNeed.jobs: List[JobBase] → ExplainableObjectDict[JobBase], same attribute name (edge/recurrent_server_need.py).

The two jobs conversions replace the duplicate-entry idiom (.jobs.count(self) in job.py) with an explicit count, so all JobBase containers share one multiplicity mechanism.

Keeping the attribute names means iteration sites (for uj_step in self.uj_steps) and the interface's attr-name-keyed UI config keep working unchanged — dicts iterate over keys; only list-specific operations (concatenation, .count, indexing) are adjusted. Python dicts preserve insertion order, so step ordering (which drives occupancy-window delays) survives the change. One source of truth per relationship — the lists are replaced, not kept in parallel.

Constructors accept sugar, normalized by one small shared helper into an ExplainableObjectDict of dimensionless quantities: a plain list (each entry weight 1, duplicates accumulating — [job1, job1] → {job1: 2}) or a dict with plain-number values ({job1: 3}). Plain numbers are wrapped by the helper as SourceValue(n * u.dimensionless) — whose source defaults to Sources.HYPOTHESIS — so weights carry the same provenance as any hand-declared input and show up normally in explanations and the sources table. ExplainableObjectDict.__setitem__ stays strict (it also backs calculated dicts, where a raw number must remain an error); wrapping happens only at the constructor boundary. EdgeDeviceGroup adopts the same normalizer. Weight labels are static ("Times per journey" / "Times per step"), without embedding object names — weights are inputs, and input labels are not recomputed when an object is renamed (same convention as user_time_spent's "Time spent by user"); the owning objects provide the context in explanations. The sugar is __init__-only: after construction the attribute is a dict, with no live-list proxy. It does not replace the JSON upgrade handler, because from_json_dict bypasses __init__ (it builds via __new__ + per-attribute __setattr__, where an old list-shaped payload hits the generic list branch); pre-feature JSONs are converted by a version-upgrade handler, the established mechanism.

On the interface side, the strategy is to generalize the edge-group dict-relationship machinery from EdgeDeviceGroup-only to any ExplainableObjectDict relationship, then apply it to journeys/steps. Three pieces exist and are currently hard-coded to edge groups: the inline canvas count partial (group_entry_count_unlink.html with hx-post update-dict-count on change), the child-panel membership section (group_membership_section.html with link/unlink/count edit), and the HTMX endpoints (views_dict_mutation.py, whose _load_group_and_key / _resolve_target_dict_attr assume EdgeDeviceGroup). Generalizing them gives steps and jobs the canvas inline editing and child-panel membership for free, and any liftup along the way lands on the edge surfaces too. The generalized resolution is annotation-driven: the (parent class, child class) → dict attribute mapping is derived from the library's __init__ annotations (ExplainableObjectDict[X]), the same single source of truth the form generator already introspects — computed once and cached, with a unit test over all modeling classes. Relationship wording (count label, membership-section title, "Add to…" button) lives in the existing field_ui_config.json / FieldUIConfigProvider, keyed by attribute name. No new registry or lookup table. The parent-panel form field reuses the existing dict_count widget.

3. Cross-cutting concerns

  • Tests (library): unit tests for weighted duration, composed multipliers (job × step), recurrent-server-need job multipliers, per-relationship independence, 0-weight, validation rejection; round-trip to/from JSON; upgrade-handler test with a pre-feature fixture containing duplicate job entries. Integration test asserting unchanged system results when all multipliers are 1.
  • Tests (interface): unit tests for form generation/parsing of the two new dict fields and for the generalized dict-mutation attribute resolution; integration tests for the three edit surfaces (inline count post, parent panel save, child membership link/unlink); e2e coverage by extending the existing edge-group inline-count e2e pattern to a journey canvas flow (inline editing on the main canvas is a critical flow per constitution §2.1) — plus a regression run of the existing edge-group tests, which exercise the same generalized code paths.
  • Migrations: no Django model change (model state is session/JSON). JSON format version bump + upgrade handler in the library covers exports and sessions.
  • Docs: library param_descriptions drive generated docs; CHANGELOG in both repos; mention the "list relationship → weighted dict" pattern in the affected AGENTS.md/CLAUDE.md per workspace policy.
  • Release ordering: library PR → PyPI release → interface PR pins the new version (constitution §2.4 gate).

4. Risks

  • Occupancy-window semantics with fractional weights. UsageJourneyStep.hourly_avg_occurrences_per_usage_pattern tiles journey occupancy by accumulating step delays; weighted times must keep the telescoping property (sum of step windows = journey duration). Mitigation: the weighted duration is the sum of weighted step times by construction; add an explicit test for the tiling invariant.
  • ExplainableObjectDict as a relationship carrier outside the edge module. Container registration, modeling_objects_whose_attributes_depend_directly_on_me, and ancestor-graph wiring must behave like lists. Mitigation: verify against EdgeDeviceGroup's wiring early; this is the first task's test gate.
  • Wide construction-site churn in library defaults, tutorials, and both repos' tests. Mitigation: mechanical, but schedule as its own task; no-behavior-change check via the all-multipliers-at-1 integration test.
  • dict_count widget ordering for steps — the widget re-renders selected rows in alphabetical option order, which would scramble step order. Mitigation: render rows by iterating the selected map (insertion-ordered) instead of the options list, for all dict fields.

5. Alternatives considered

  • Keep the lists and add parallel weight dicts. Preserves current list machinery but creates two sources of truth that must stay in sync on every add/remove — rejected for leanness (no external callers to protect).
  • Multiplier as a property of the step/job object itself. Rejected: reuse across parents requires per-relationship values (spec §2).
  • Wrapper objects ("WeightedStep") in the lists. A new modeling-object layer per entry — heavier in serialization, UI, and explainability than a quantity-valued dict; rejected.
  • Weighted-list literals ([ExplainableQuantity(2 * u.dimensionless) * job1, …]). Overloading multiplication to pair quantities with modeling objects adds API surface for little gain — the dict-with-plain-numbers sugar already covers dev ergonomics, and most models are built through the interface or AI assistance; rejected.
  • __setattr__-level list→dict coercion to avoid the JSON upgrade handler. Would let old list-shaped JSONs load with no handler (the only path that could, since from_json_dict bypasses __init__), but puts annotation-aware coercion in core attribute-setting machinery and makes the legacy JSON shape implicitly valid forever; a ~15-line upgrade handler is cheaper and explicit. Rejected.

6. Constitutional notes

  • §1.3 (library is domain truth): all semantics, validation, and explainability land in efootprint; the interface only edits/displays — respected.
  • §1.1 (clean architecture): new form fields flow through the existing annotation-driven generator; no domain → adapter leakage expected.
  • §2.4 (PyPI before main): release-ordering noted in §3.
  • §4 (no backward-compat shims): the JSON version-upgrade handler is the established versioning mechanism, not a shim; Python-API compatibility is deliberately broken.
