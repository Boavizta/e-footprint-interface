Step 6 — Introductory templates + guided tour — Implementation plan

Status: Plan — approved · 2026-05-29

Spec: spec.html. Sources: 02-interface-onboarding-ux.md, 04-efootprint-docs-overhaul.md, 99-roadmap.md (Step 6). Respects specs/constitution.md and specs/architecture.md.

1. Approach

Step 6 turns the blank-canvas first run into a guided one, in three sub-phases that ship and review independently but form one capability. Sub-phase A adds an interface-owned modeling_templates/ tree: three serialized System JSONs (ecommerce, ai_chatbot, iot_industrial) plus a typed registry.py of UI metadata, and an empty-but-wired other/ sibling. The current default_system_data.json is slimmed to a truly empty System and repurposed as the “Start from scratch” baseline. Sub-phase B adds a server-rendered template picker partial that merges those introductory templates with the library’s how-to templates (consumed at runtime via efootprint.modeling_templates.list_how_to_templates()) and a first-class “Start from scratch” option; the picker overlays the canvas whenever the builder is entered with an empty model, and is reachable any time from a new help menu in the toolbar (next to the edge toggle). Picking a card loads that system into the session and swaps the canvas. Sub-phase C adds the guided tour — a thin tour.js wrapper around driver.js (a small, dependency-free, MIT tour library) driven by server-provided step copy and targets, with two flavors (real targets on a loaded template, placeholders + a “create a usage journey” suggestion on a blank start). The tour auto-runs once ever (a localStorage flag) and is replayable from the help menu.

Key principle preserved throughout: the state-bearing parts stay HTMX/server-rendered (picker partial, template load → canvas swap, help drawer, first-run trigger computed server-side), while only the inherently client-side view logic (highlighting, tooltip positioning, step sequencing, non-blocking overlay) is JS — the constitution’s sanctioned “vanilla JS where unavoidable,” not a deviation (§5 of the spec, constitution §1.4).

Key abstractions / entry points

  • model_builder/domain/reference_data/modeling_templates/introductory/registry.py — typed IntroTemplate metadata mirroring the library’s HowToTemplate (id, name, description, icon, showcased_concepts tokens, json_path, category, optional tags), plus a closed CONCEPTS enum (token → label + optional help/doc target) for concept chips that map to no single efootprint class.
  • A new onboarding view module (adapters/views/views_onboarding.py) — picker partial, template-load endpoint, and tour-steps payload.
  • A merged-template service in the domain that combines the introductory registry, efootprint.modeling_templates.list_how_to_templates(), and the (empty) other registry into category groups for the picker.
  • theme/static/scripts/tour.js (IIFE + data-action) wrapping driver.js; step copy resolved server-side through the existing interface_placeholder_handlers ({kind:target} → HTML).

3. Cross-cutting concerns

  • Tests.
    • Unit — registry metadata consistency (every json_path exists; category valid; showcased_concepts tokens resolve through the placeholder registry; icon present). Mirrors the library’s HowToTemplate validation.
    • Unit/integration — load-and-compute per introductory template: each JSON loads via json_to_system / ProgressiveImportService and produces computed emissions without error. Pattern from e-footprint/tests/test_modeling_templates.py and the interface’s test_progressive_import_service.py. Fails if a template stops loading/computing (spec success criterion).
    • Integration — the catalog service merges all three categories (introductory + library how-to + other), runs adapters→use cases→domain with no Django scaffolding (constitution §1.1).
    • E2E (Playwright) — the critical first-run flow only: enter with empty model → picker appears → pick a template → canvas loads + tour runs; replay from help menu re-opens the tour; IoT template lands with the edge toggle latched on; “Browse templates” CTA on home opens the picker. Use click_and_wait_for_htmx(); build fixtures with from_defaults() (testing.md).
    • Jest — onboarding_first_run.js flag logic and tour.js step wiring (mock driver.js).
  • Migrations. None. First-run state is localStorage-only — no Django model, no migration (spec §6, constitution §2.5). Worth stating explicitly so reviewers don’t expect one.
  • Docs. Update specs/architecture.md (new interface-owned modeling_templates/ tree + registry pattern; the empty-model picker trigger; the driver.js tour engine and its server-provided-steps contract), specs/conventions.md (driver.js as the sanctioned dependency-free tour lib; where tour copy lives), specs/tech_stack.md (driver.js version bound). Add a CHANGELOG.md entry (constitution §2.6).
  • Dependency hygiene. efootprint may be an editable path during co-development but must reference PyPI in pyproject.toml/poetry.lock before reaching main (constitution §2.4) — the how-to templates ship with the installed package.
  • Deep-link targets. Any mkdocs link the picker/tour relies on must point at a page delivered in Step 4 (how-to pages) via the configured mkdocs_base_url (spec §5).

4. Risks

  • Non-blocking overlay vs. help drawer z-index. driver.js’s overlay must not trap the user — the help drawer (#helpDrawer, higher z-index sibling) has to stay clickable while the tour is active. Mitigation: configure driver.js with click-through/no-modal behavior on the help step and assert drawer interactivity in E2E.
  • “Empty model” predicate drift. The predicate is fixed (only efootprint object key is System; see §7) and unit-tested. Residual risk: the excluded-metadata-key set (efootprint_version, Sources, interface_config, efootprint_interface_version) must stay in sync with the serializer if new top-level metadata keys appear. Mitigation: derive the exclusion from the serializer where possible, and assert it in the predicate unit test.
  • Tour selector brittleness. Targets coupled to incidental DOM break when the canvas markup changes. Mitigation: stable data-tour-target attributes, asserted present in E2E.
  • IoT edge-latch dependency. Relies on the Step 5 toggle/latch firing from loaded edge objects — verified present (modeling_paradigm.py, ModelWeb.has_edge_objects, oob_regions.py::edge_modeling_toggle). Risk only if Step 5 regresses; covered by the IoT E2E assertion.
  • Mobile fidelity. Tooltip anchoring against a horizontally-scrolling canvas on a narrow viewport may cost complexity out of proportion to value. Per spec §5/§6 and constitution §4, this is raised for discussion at implementation time, not silently dropped or over-built.
  • Default-data slimming. Existing tests/fixtures may assume the seeded default journey/step. Mitigation: grep for dependencies on the old default and update them as part of Sub-phase A (no papering over — constitution §3.1).

5. Alternatives considered

  • Hand-rolled vanilla tour engine — rejected (plan-time decision): more code to maintain for the fiddly tooltip-anchoring and reposition-on-scroll/resize logic the spec calls out. driver.js is small, dependency-free, non-SPA, and keeps custom JS minimal — closer to the “minimize JS” intent.
  • Heavier tour libraries (Shepherd.js, intro.js) — rejected: larger footprint and/or extra deps for no added value over driver.js.
  • Picker as a separate page/route — rejected: the spec mandates no new pages or navigation; everything new is an overlay on the single canvas (spec §4).
  • Picker as a Bootstrap modal or side-panel slide-in — rejected in favor of a bespoke canvas overlay: the modal’s dialog sizing and the side panel’s narrow slide-in both fight the full-width multi-group welcome layout; the bespoke overlay also swaps cleanly via HTMX and keeps the app chrome framing the canvas.
  • A new {concept:X} placeholder kind — rejected for now: a closed CONCEPTS enum local to the registry covers the picker chips without touching the shared placeholder resolver. The placeholder kind can be added later if concept tokens are wanted inside tour copy or help text.
  • First-run state in user profile + localStorage fallback (floated in topic 02) — rejected: spec §6 simplifies to localStorage-only (no Django model, no migration). Trade-off: onboarding memory does not follow a signed-in user across browsers/devices — accepted.
  • Keep default_system_data.json as a seeded skeleton — rejected: a non-empty default would make the blank tour’s “create a usage journey” suggestion incoherent (spec §6).

6. Constitutional notes

  • §1.1 Clean Architecture. Registry + template loading are domain data/services; the catalog service runs without Django; views stay thin adapters; the picker partial and tour-steps rendering are presentation. Tour copy resolution reuses the existing DescriptionProvider/placeholder port. No Django imports leak into domain/.
  • §1.3 Library is the truth. How-to templates are consumed from the library’s public API at runtime; the interface authors only the introductory/other registries and never re-implements modeling.
  • §1.4 HTMX-first; minimize JS. Picker, template-load→canvas swap, help drawer, and the first-run trigger are server-side. The driver.js tour engine is view-only logic with no domain state and no routing — the sanctioned “vanilla JS where unavoidable,” explicitly not a deviation (spec §5). No client-side data layer shadows the server.
  • §2 Quality gates. Template load+compute+metadata tests, a CHANGELOG.md entry, and PyPI-referenced efootprint before main are in scope; localStorage-only state means §2.5 (migration) does not apply.
  • §4 Out of scope. Desktop-first respected; mobile is best-effort and explicitly flagged for implementation-time discussion. No i18n, no SPA framework (driver.js is not one).
  • No amendment required. Adding driver.js is a tech_stack.md/conventions.md addition, not a constitutional change. (Confirm: §4 bans “replacing Django/HTMX/Bootstrap/Playwright,” not adding a small view-only utility — driver.js does not replace any of them.)

7. Resolved decisions

The open questions from the spec and the first plan draft were resolved with the user as follows. None remain blocking; tasks.md can be emitted next.

  • Empty-model predicate. The picker triggers when system_data is None, or when the serialized JSON’s only efootprint object key is System — metadata keys (efootprint_version, Sources, interface_config, efootprint_interface_version) excluded. Robust to the empty System carrying empty usage_patterns / edge_usage_patterns lists. Unit-tested.
  • Template authoring. A committed Python scenario constructor per template is the source of truth; a one-shot script (scripts/build_intro_templates.py) regenerates the JSONs via system_to_json, so they can be re-built when the library schema changes.
  • Picker overlay. A bespoke canvas overlay (HTMX partial swapped into a canvas-scoped container) — not a Bootstrap modal or side panel. Best fit for the full-width welcome layout, clean HTMX swap, app chrome stays framed; dismissal and responsiveness are hand-wired.
  • mkdocs deep-links. Included now: how-to picker cards (and the tour’s help step) deep-link to the specific Step 4 how-to pages via {doc:…} / mkdocs_base_url, gated on confirming those pages are live.
  • Showcased-concept tokens. A closed CONCEPTS enum in the interface registry (token → display label + optional help/doc target) combined with {class:X} tokens where a real efootprint class fits. CI-validated; the shared placeholder resolver is left untouched (no new {concept:…} kind). Concepts are picker-card metadata only — the tour does not consume them.
  • Mobile. Desktop-first, mobile best-effort: a responsive card grid and driver.js default mobile positioning; disproportionate cases (tooltip anchoring against the horizontally-scrolling canvas) are raised at implementation time rather than silently dropped or over-built.

8. Sequencing

Sub-phases ship in order, each gated by review (roadmap Step 6): A (templates + registry; the data layer everything else consumes) → B (picker + help menu + first-run trigger; depends on A and on the library how-to API) → C (tour; depends on B’s help-menu replay entry point and canvas targets). Each gets its own task list under this feature folder; a single tasks.md may group them with per-sub-phase sections.
