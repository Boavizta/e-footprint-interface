Ordering rationale

The three tasks are the plan's three sub-phases (plan §8), and the ordering is forced by their data/UI dependencies:

  • Task 1 before 2 — the picker and catalog service consume the introductory registry, the generated JSONs, and the slimmed empty baseline. Until they exist there is nothing to merge or to load.
  • Task 2 before 3 — the tour's replay entry point lives in the help menu and its targets live on the loaded canvas; both are delivered in Task 2. The tour also rides on the first-run state plumbing Task 2 introduces.

Why each sub-phase is one task, not several. Within a sub-phase there is no behavioural pause point where the system is both shippable and meaningfully demoable as a standalone increment — the units are tightly coupled toward a single milestone:

  • Task 1: the registry, the JSONs it points at, the build script that regenerates them, the default slimming, and the metadata/load-compute tests that guard them are one consistency boundary — splitting "JSONs" from "the registry that validates them" or from "the test that asserts they load" creates review boundaries with no working pause between them.
  • Task 2: the catalog service, picker partial, load endpoint, empty-model trigger, help menu, home CTA, and first-run flag together deliver exactly one observable behaviour ("first-run picker appears and loads a system"); each piece is inert without the others (a picker with no trigger, a trigger with no partial, a load endpoint with no card to invoke it).
  • Task 3: the driver.js dependency, tour.js, the server-rendered step payload, and the canvas data-tour-target anchors are a single feature — the engine is useless without targets and the targets without the engine.

Each task leaves the system in a working state with its suite green, so each is independently shippable as one PR and reviewable in one pass.
