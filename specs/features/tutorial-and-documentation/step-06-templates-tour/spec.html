Step 6 — Introductory templates + guided tour

Status: Spec — under review · 2026-05-28

Part of the tutorial-and-documentation overhaul (roadmap Step 6). Draws on topic files 02-interface-onboarding-ux.md and 04-efootprint-docs-overhaul.md.

1. Problem and audience

  • Problem. A first-time visitor lands on a blank model builder with no guidance: no example to mutate, no orientation to where things live, no explanation of the order to model in. The interface has contextual help (info icons, help drawer) and disabled-instead-of-error buttons after earlier steps, but nothing that greets a new user or gives them a running start. For the primary audience this is the difference between "I get it" and "I close the tab."
  • Audience. Primarily the non-technical product person with zero Python and zero modeling background; secondarily technical users who want to sense depth and drop into it. Frequency: the first run is once-per-user but decisive; the replay paths serve returning users who want to start over or explore a different shape of system.
  • Why now. The prerequisites are in place: contextual help and disabled-UX (Steps 1–3), the library’s how-to modeling templates exposed as a public API (Step 4), and the edge modeling toggle + latch (Step 5). Onboarding is the payoff that ties those investments into a coherent first-run experience — and it is the last user-facing piece of the overhaul before the final build review.

2. Success criteria

Each criterion is phrased as a user-observable, testable outcome.

  • The existing marketing home page is preserved as the front door; it is not skipped on a first visit. It gains one addition: a secondary “Browse templates” CTA beside “Start modeling yourself” that opens the picker directly. Onboarding otherwise triggers at the model-builder entry — so “first run” means first entry into the model builder, not first hit on the site root.
  • Whenever the builder is entered with an empty model (the first-ever entry, or a later return by someone who never built anything), the user is met with a template picker instead of a blank canvas. It offers a small set of ready-to-mutate introductory systems, the library’s how-to example systems, and a first-class “Start from scratch” option (which loads a truly empty system). Once the model has content, entry goes straight to it.
  • Choosing any introductory or how-to template loads a complete, working system the user can immediately edit, whose results compute without error.
  • Choosing the IoT/industrial template lands the user in a model where edge modeling is already active (the Step 5 toggle is latched on by the loaded system), with no separate “web or edge?” prompt.
  • After a template loads (or after “Start from scratch”), a short guided tour orients the user to the interface map (where usage journeys, infrastructure, usage patterns, and results live), the recommended modeling order, and where contextual help is found. The tour is dismissible at any step.
  • The tour teaches concepts by pointing at the existing help affordances — the help buttons next to the add buttons, the field tooltips, the info icons — not by carrying its own concept text. At the help step it can open the help side panel itself so the user sees it populated.
  • The tour is non-blocking: the user can open and explore the help side panel while the tour is still active and then continue, rather than being trapped in a modal overlay.
  • The tour auto-runs only on a user’s first-ever visit. It does not reappear on later visits.
  • A user can replay the tour and re-open the template picker at any time from a persistent help entry point, without losing the ability to keep their current work if they choose not to switch.
  • The tour points at real on-screen targets when a template is loaded, and at placeholder targets with a concrete first-action suggestion when the user started from scratch.
  • The tour teaches the map and order only; domain concepts are not duplicated into tour copy — they remain in contextual help and mkdocs deep-links.
  • Every shipped template and the metadata that describes it are covered by tests that fail if a template stops loading, stops computing, or its metadata becomes inconsistent.

3. Scope

Capability-level only. File paths, registry/module names, the JSON storage scheme, the guided-tour mechanism, first-run state plumbing, and the fate of the current default-system data belong in plan.html (and in the existing topic files 02/04), not here.

Step 6 is delivered in three sub-phases, sequenced by their prerequisites. They form one user-facing capability — the first-run experience — and are scoped together here even though each gets its own implementation plan.

In scope
  • Sub-phase A — Introductory templates. A small curated set of ready-to-mutate example systems (a classical e-commerce service, an AI chatbot, an industrial IoT setup) plus the descriptive metadata each needs to appear in the picker (display name, short description, an icon, and the concepts it showcases). The IoT template demonstrates edge modeling in context. A wired-but-empty home for ad-hoc “other” examples so they can be added later without new plumbing.
  • Sub-phase B — Template picker. A picker screen offering the introductory templates, the library’s how-to example systems, and “Start from scratch”, shown whenever the builder is entered with an empty model and reachable any time from the help entry point. A “Browse templates” CTA on the home page opens it directly. A persistent help entry point in the navbar toolbar lets a user replay the tour or re-open the picker mid-session. Onboarding-seen state is tracked in localStorage (per-browser; no user-profile flag, no Django model).
  • Sub-phase C — Guided tour. Two flavors sharing the same orientation content: one overlaid on a loaded template pointing at real cards/panels, and one for the blank start that uses placeholders and suggests the first concrete action. Tour scope: interface orientation, recommended modeling order, and contextual-help discoverability — including a step that opens the help side panel so the user sees it in use. The overlay is non-blocking, so the user can explore the help panel alongside the running tour. The tour auto-runs on first-ever visit and is replayable from the help entry point.
Out of scope (this iteration)
  • Re-authored domain prose in the tour. The tour carries no concept explanations of its own and embeds no copied help text. It teaches concepts only by pointing at — and opening — the existing SSOT-fed help affordances (help buttons, field tooltips, info icons, help drawer) and mkdocs deep-links. Surfacing that content is in scope; duplicating it into tour copy is not.
  • Step-by-step wizard or video onboarding. Explicitly rejected in topic 02.
  • Redesigning the marketing home page. Its only change is the added “Browse templates” CTA (in scope, above); the page is otherwise preserved as-is. No layout, hero, or section rework.
  • New library how-to templates. Step 6 consumes the how-to templates delivered in Step 4; it does not author more.
  • A full multi-category template browser beyond first-run. First-run shows introductory + how-to + scratch; a richer merged browser is a later concern. The “other” category ships only as an empty, wired placeholder.
  • Persistent per-user model libraries and analytics/telemetry on tour completion or template choice (out of scope per mission unless a telemetry story already exists).
  • Authoring a new contextual-help content store. Tour copy is interface-specific, but contextual help still draws from the SSOT plumbing from Steps 2–3; no parallel content store is introduced.

4. UX / interface

The interface is not getting new pages or new navigation. The existing marketing home page stays as the public front door; its only addition is a secondary “Browse templates” CTA beside “Start modeling yourself”. The model builder is one canvas — columns of cards, a right-side help drawer and edit panel that slide in, and a results bar at the bottom. Everything new is an overlay on that same canvas: the picker opens over it, the tour points at its cards, the help menu lives in its chrome.

The home page is not short-circuited on a first visit — a cold visitor still gets the marketing context that helps them decide to engage. The picker then appears whenever the builder is entered with an empty model (the first-ever entry, or a later return by someone who never built anything) — so “first run” is anchored on the model being empty, not on first hit of the site root. Once the model has content, entry goes straight to it. The tour auto-runs once ever (a localStorage flag); both picker and tour are reachable any time from the help menu.

5. Constraints

  • Mission fit. Onboarding with starter templates, contextual help, and the web/edge toggle is named as forthcoming scope in mission.md; the primary audience (non-technical product person, zero Python) governs tone and defaults.
  • Constitution — HTMX-first, minimize JS. The rule is the boundary, and this feature splits cleanly across it. The state-bearing parts stay HTMX/server-rendered: the picker is a server-rendered partial whose selection loads a System into the session and swaps the canvas; the help drawer is already an HTMX endpoint; first-run state is a single server write; tour step copy and targets are server-provided so the words don’t live in JS. The tour engine (element highlighting, tooltip positioning, step sequencing, reposition on scroll/resize, non-blocking overlay) is inherently client-side view logic with no domain state and no routing — it is precisely the constitution’s sanctioned “vanilla JS where unavoidable,” not a deviation from it. “Minimize JS” here means “no client-side data layer shadowing the server,” which the tour respects. A small dependency-free tour library is permissible (it is not an SPA framework); hand-rolled vanilla vs. such a library is a plan-time maintainability call, not a constitutional one.
  • Constitution — Clean Architecture. First-run state, template loading, and tour orchestration must respect the domain/application/adapter dependency direction; the library stays the modeling truth and the interface never re-implements modeling.
  • Constitution — quality gates. Templates and their metadata need test coverage (load + compute + metadata consistency). Onboarding-seen state is localStorage-only, so it adds no Django model and needs no migration; a CHANGELOG.md entry is still required, and efootprint must be referenced from PyPI (not an editable path) before reaching main.
  • Cross-step dependencies. Sub-phase A’s IoT template relies on the Step 5 edge toggle/latch (verified present). Sub-phase B surfaces the library’s how-to example systems through the public API delivered in Step 4 (verified present in the installed library). Both prerequisites are satisfied.
  • SSOT, not a parallel store. Contextual help surfaced during onboarding continues to draw from the Step 2–3 description plumbing; only the tour’s own step copy (interface-specific, not derivable from library classes) is new content.
  • Deep-link targets must exist. Any mkdocs deep-link the tour or picker relies on must point at a page delivered in Step 4.
  • English only per mission and constitution.
  • Desktop-first, but should also work on mobile. Desktop is the primary target (per mission/constitution), yet the picker and tour should remain usable on a small screen — mobile is not to be ignored. Where genuine mobile support would add complexity out of proportion to its value (e.g. tooltip anchoring against a horizontally-scrolling canvas on a narrow viewport), that is to be raised and discussed at implementation time rather than silently dropped or silently over-built.

6. Decisions and remaining questions

The questions that were open at first draft are now resolved as follows:

  • “Start from scratch” loads a truly empty system — a named System with no usage journeys, servers, or patterns. The pre-seeded journey/step in today’s default is dropped, so the blank tour’s “create a usage journey” suggestion stays coherent.
  • Fate of the current default system data follows from the above: it becomes the empty “Start from scratch” baseline (slimmed to an empty system). It is not kept as a seeded skeleton and not promoted into the introductory set.
  • The help entry point lives in the navbar toolbar, next to the edge-modeling toggle, where global model-level controls already sit. It follows the same desktop/mobile merge behavior as the rest of that toolbar.
  • Showcased concepts are tokens resolved through the placeholder registry to canonical class/concept names — CI-validated, no free-form drift.
  • The blank tour’s first concrete action is “create a usage journey”, matching the recommended modeling order the tour teaches.
  • The home page gains a secondary “Browse templates” CTA next to “Start modeling yourself”, opening the picker directly. It remains otherwise unchanged as the front door.
  • Onboarding-seen state is stored in localStorage only — per-browser, no user-profile flag and no Django model (hence no migration for first-run state). This is a deliberate simplification over the profile+fallback split floated in topic 02; it means onboarding memory does not follow a signed-in user across browsers/devices.
  • The picker auto-appears whenever the builder is entered with an empty model — the first-ever entry is simply the first instance of that, and a returning user who never built anything is re-offered it. Once the model has content, entry goes straight to it; the picker is then reachable only via Help ▸ Open templates. Independently, the tour auto-runs once ever (gated by the localStorage flag) and is otherwise replayed via Help ▸ Replay tour.

Genuinely deferred to implementation time (not blocking the plan):

  • Tour engine: hand-rolled vanilla vs. a small dependency-free tour library — a maintainability call, both constitution-compliant (see §5).
  • Mobile fidelity of the picker and tour — usable on small screens is the goal; where it would cost disproportionate complexity, raise it for discussion rather than silently dropping or over-building (see §5).
