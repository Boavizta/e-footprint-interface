Model comparison — Implementation plan

Status: Plan — under review · 2026-06-16 · Spec: spec.html (rev 3, v1) · Backlog: backlog.html
Cross-repo: computation core in e-footprint (library), workspace/UI in e-footprint-interface · home stays here (interface-driven); one tasks.md covers both.

1. Approach

The session moves from holding one model to holding a small workspace of up to two model slots plus an active-slot pointer. Each slot keeps its own cached blob (today's exact single-model payload), so editing one model never deserializes or recomputes the other — the binding Redis-RAM / JSON-round-trip constraint stays respected. The 30 MB Redis cap (MAX_PAYLOAD_SIZE_MB) becomes a shared budget across both slots' with-calculated-attributes payloads (the real Redis footprint), not a per-slot allowance. A new slot-aware workspace repository owns the slot index and vends an ordinary per-slot ISystemRepository; because that per-slot repository defaults to the active slot, the ~25 existing ModelWeb(repository) call sites keep working unchanged — only the new comparison view explicitly addresses both slots.

The builder UI gains a tab strip (Model A · Model B · ＋ Add · ⇄ Compare). Both model canvases are rendered server-side and kept resident in the DOM, toggled client-side — the same lazy-render-then-show/hide pattern the Results/Sources tabs already use, so switching models costs no network round-trip and preserves each canvas's transient UI state. Only the active canvas is editable; all chrome (side panel, help drawer, toolbar, results panel) stays a single shared instance rebound to the active slot. DOM-id collisions between the two resident canvases are avoided at a single chokepoint — the wrappers' web_id property (every card id, derived id and leaderline anchor flows from it) — by prefixing it with the system id; the canvas templates need no edits, and server routing is unaffected because it already uses a separate efootprint_id. The underlying efootprint object ids are preserved through duplication (a duplicate gets a fresh system id only), which is what lets the comparison diff pair objects by identity. The workspace keeps the two slots' system ids distinct as a load-bearing invariant — minting a fresh one on any collision (re-imported file, same template loaded twice) — since the web_id prefix depends on it. The Compare tab is re-rendered fresh on every visit (read-only, cheap), guaranteeing "no stale results" without keeping comparison DOM warm.

The comparison computation lives in e-footprint, not the interface (constitution §1.3 — the library is the domain truth). A new efootprint.comparison capability — surfaced as system_a.compare_to(system_b) returning a SystemComparison — computes the KPI totals + deltas, the per-(category, phase) decomposition (from the library's own total_*_footprint_sum_over_period dicts, which already sum to the total), the aligned/​cumulative time-series, and the input diff (id-first, name+type fallback). It also ships simple matplotlib/plotly plots so the comparison is useful directly in a Jupyter notebook or to a coding agent — even though the web re-renders them. This makes the feature cross-repo: the interface side shrinks to a thin adapter (ComparisonService) that calls the library and shapes its result into Chart.js JSON + the HTML diff table — it re-implements rendering, never the modeling. No new modeling logic, no attribution claims.

2.7 Workspace file format

The workspace file is a thin envelope wrapping two byte-for-byte single-model documents — so the single-model format is untouched and the per-model import/serialization path is reused per element, not re-implemented:

Envelope shape 
{
  "efootprint_workspace_version": "<interface version>",
  "active_slot": 0,
  "models": [
    { /* slot 0: exact single-model export doc — efootprint_version, Sources,
         <object types>, interface_config, efootprint_interface_version */ },
    { /* slot 1: exact single-model export doc */ }
  ]
}
  • Detection is content-based (authoritative): a top-level models key ⇒ workspace file; its absence ⇒ single-model file. A distinct .e-fw.json extension (vs. today's .e-f.json) is a UX hint only.
  • Cross-format robustness (honest-error / disabled-instead-of-error): "Open workspace file" fed a single-model file loads it into the active slot; "Replace this model" fed a workspace file errors with guidance (or offers to open it as a workspace) rather than corrupting state.
  • No calculated attributes in the file (each element is the without-calc single-model export); they are recomputed on import, and only then does the shared budget apply.
  • Each model carries its own interface_config inside its element — no separate handling, Sankey settings survive the round-trip per model as today.
  • Filename: "{datetime} UTC workspace ({name A} vs {name B}).e-fw.json" (sanitized/truncated like the single-model name).

Keep the Clean-Architecture no-Django-in-domain guard green.

4. Risks

  • Duplicate DOM ids across resident canvases. The headline correctness risk. Mitigation: system-id prefix on the ModelingObjectWeb.web_id chokepoint + preserved object ids + fresh system id per slot; an E2E/Jest assertion that no id appears twice across the two canvases.
  • Two slots with the same system id. The web_id prefix scheme above is only safe while the two slots hold distinct system ids — but they easily won't: re-importing the same/lineage file into the second slot, loading the same template into both slots (templates carry baked, stable ids), or a workspace file whose two models share an id. Same id ⇒ identical web_id prefixes ⇒ the collision returns. Mitigation: a single distinct-system-id invariant enforced in the workspace repository's add-to-slot path (§2.1) — mint a fresh system id on any cross-slot collision, regardless of source. Diff-safe because the comparison matches object ids, not the system id. Covered by a "import the same file/template into both slots" integration test.
  • OOB swaps targeting the wrong canvas. HTMX OOB resolves first-match-by-id, so the model_canvas region (oob_regions.py) must emit the active slot's #model-canva-{slot}. Shared chrome (results buttons, edge toggle) is a singleton, so it is unaffected. Covered by a mutation-on-B-while-A-resident integration test.
  • Call-site sweep regressions. ~25 ModelWeb(repository) sites change. Mitigation: the per-slot repo defaults to the active slot so behaviour is identical in single-model sessions; the existing suite is the regression net; do it as one mechanical sweep, not ad-hoc.
  • Leader-line work for the hidden canvas. Lines must not be computed for the parked tab. Mitigation: build/teardown keyed on tab visibility; the existing "rebuild on shown/hidden" hooks extend to tab switch.
  • Payload / memory roughly doubles. Two slots cached + two canvases resident. Mitigation: per-slot blobs mean only the active model is (de)serialized per edit; the MAX_PAYLOAD_SIZE_MB cap is enforced as a shared budget on the summed with-calc (Redis) weight of both slots (not per slot, which would double the real footprint); hidden DOM is cheap (no layout/paint).
  • Duplicate / import can blow the budget instantly. Duplicating a large model roughly doubles Redis use in one action. Mitigation: the ＋Add (duplicate) and second-slot import paths pre-check the prospective summed size and refuse with a clear "the two models together would exceed N MB" message before persisting — ideally surfaced as a disabled ＋Add option when the active model already nears half the budget.
  • Diff "only in A/B" noise for independently-built models. id-first matching falls back to (name, type); two from-scratch models still diff sensibly, but unrelated objects will read as A/B-only — expected, not a bug.
  • Cross-repo version coupling. The interface depends on the new library release; until it is on PyPI, interface work runs on the editable local install and must not reach main with that path. Mitigation: the §3 sequencing; the existing pyproject-parsing test guards the editable path.
  • Taxonomy drift between comparison and single-model results. The library comparison derives categories from OBJECT_CATEGORIES; the interface's single-model results still use EmissionsCalculationService's hand-rolled keys. Mitigation: align the comparison's category labels to what the results panel already shows; consolidate the interface onto the library taxonomy later (out of v1 scope).

5. Alternatives considered

  • Single combined session blob {models:{a,b}, active} — rejected: every request would (de)serialize both models and rewrite the larger blob on each edit, against the binding perf constraint. Per-slot keys load/save only the edited model.
  • Keep the legacy cache key for slot 0 (asymmetric) — rejected for long-term maintainability: a permanent special case in the repository. Symmetric keys + a self-deleting one-release fallback give clean addressing with no in-flight data loss.
  • Full-content HTMX swap on every model switch — rejected: re-renders the builder, makes a network call, and loses transient UI state on each switch. Resident tabs (the existing Results/Sources pattern) avoid all three; the only added cost is id-namespacing, which the system-id prefix handles.
  • Re-id objects on duplicate — rejected: it would destroy the shared object identity the comparison diff relies on. Preserving object ids (new system id only) keeps id-based pairing intact.
  • Name+type-only diff matching (as the spec literally states) — superseded: id-first is robust to renames in the duplicate-then-edit workflow; name+type is kept only as the fallback for independent models.
  • Keep the Compare dashboard resident/cached — rejected: violates "no stale results." It is cheap to re-render on each visit (data already computed per model).
  • Comparison logic in the interface domain (original draft) — superseded: putting the computation in efootprint keeps it usable from a notebook / coding agent and honors §1.3 (library is the domain truth). It costs a cross-repo release cycle, but the inputs and plotting deps already exist there, so the marginal cost is low and the interface shrinks to an adapter.
  • A single merged "diff system" instead of two live Systems — rejected (mirrors backlog's rejected diff-Sankey): fragile when the two structures diverge. The comparison reads two independent Systems.

6. Constitutional notes

  • §1.1 Clean Architecture. The interface ComparisonService (thin adapter) and the workspace port live in domain/ with zero Django imports; slot resolution and the session index live in adapters/. The no-Django-in-domain integration guard must stay green.
  • §1.2 Repository pattern. New IWorkspaceRepository port; session and in-memory implementations stay interchangeable. ISystemRepository remains a single-model abstraction, now slot-targeted.
  • §1.3 Library is the domain truth — strengthened. The comparison computation (deltas, decomposition, input diff, duplication) and its notebook plots live in efootprint, not the interface. ComparisonService only adapts library output to the web (Chart.js JSON + HTML table). No new modeling, no attribution/sensitivity claims. This is the §1.3-correct placement and the reason the feature is cross-repo.
  • §1.4 HTMX-first, minimize JS. Resident tabs reuse the blessed Results/Sources show/hide pattern — server-rendered HTML toggled client-side, no client-side data layer, no SPA framework. The tab toggle and leader-line scoping are vanilla/hyperscript.
  • §4 Out-of-scope — not relaxed. No multi-user collaboration, no server-side library, no SPA. Therefore no constitutional amendment is required; the only doc edit is the mission.md "singular session" line (a description, not a rule).
