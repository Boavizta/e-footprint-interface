Model comparison

Status: Spec — approved, rev 4 (scoped to v1) · 2026-06-16 · §6 decisions resolved; diff matching id-first (§4.2/§4.3)

1. Problem and audience

  • Problem. The interface holds exactly one model per session. Yet the central question users bring to an impact tool is comparative: "is design B better than design A, and by how much?" — on-premise vs cloud, web vs edge architecture, current usage vs projected growth. Today, answering it means exporting one model, rebuilding or re-importing the other, and comparing numbers across separate sessions, screenshots, or xlsx files by hand. The comparison — the actual decision-support output — happens outside the tool.
  • Audience. All three mission personas. The non-technical product person gets the most value: an absolute footprint figure is hard to act on, while a delta between two concrete designs is directly actionable in a product discussion. Technical and industrial users compare architecture alternatives (e.g. fleet compositions) the same way.
  • Why now. The single-model loop is mature — build, edit, real-time results, import/export all work. Comparison is the natural next step on the decision-support ladder, and it builds purely on existing capabilities (two instances of what already exists, plus one unified view).

2. Success criteria

How do we know this feature is done? Each criterion must be testable.

  • From a normal single-model session, the user can add a second model in any of three ways: duplicating the working model, importing a previously exported model file, or starting fresh (blank or from a starter template).
  • The user can switch between the two models at any time; the active model offers the full existing builder experience (create / edit / delete objects, results panel, xlsx export) with no feature loss compared to today.
  • Each model can be exported and imported independently. A file exported from either slot opens correctly in a plain single-model session, and any pre-existing single-model export imports into either slot — full round-trip compatibility with the current file format.
  • The user can also export both models together as one workspace file, and re-import that file later to restore both models in one action.
  • Each model is identified everywhere (switcher, comparison view, file names) by the name of its modeled system — the same name the user already gives their system today. When both systems happen to share a name, the UI disambiguates them visibly.
  • A unified comparison view shows, for both models: headline totals with the difference (absolute and %), a per-category breakdown of what explains the difference, and compared versions of the emissions-over-time and cumulative-emissions charts (§4).
  • The comparison view shows what differs between the two models' inputs — changed assumption values and objects present in only one model. It shows the differences directly (no "show identical rows" mode).
  • All compared aggregates are full-timespan sums over each model's own modeling period; each model's card states its total and its period.
  • Both compared charts are magnitude-honest: the two models share scales and legends, so relative sizes can be read directly.
  • An edit to either model is reflected in the comparison view the next time it is shown — no stale results.
  • Both models, their names, and the active selection survive a page refresh within the same session.
  • The user can remove the second model and return to a plain single-model session.
  • A user who never adds a second model experiences today's flow unchanged (no regression, no new mandatory UI steps).

3. Scope

Capability-level only. File paths, class/method/parameter names, refactor instructions, internal hypotheses, and "mirror pattern X" code-level decisions belong in plan.html, not here. If you can't say it without naming a file, class, or method, it's plan content.
In scope (v1)
  • Holding two independent models in one session, one active for editing at a time, with a visible switcher.
  • Creating the second model by duplication of the first, by file import, or from blank/starter template; removing it to return to single-model mode.
  • Independent per-model import and export (existing file format, both directions), plus a combined workspace export/import restoring both models at once.
  • A dedicated full-page comparison view (the "Comparison dashboard" layout) containing: headline totals + delta; per-category delta decomposition; compared emissions-over-time chart; compared cumulative chart; and the input assumptions diff.
  • All comparison aggregates computed as full-timespan sums.
Out of scope (v1) — see backlog.html
  • Sankey repartition comparison (paired diagrams at shared flow scale).
  • Per-year / differing-timespan normalization — v1 uses full-timespan sums and a shared calendar axis only.
  • Δ-only per-period chart mode.
  • Cause linking (relating input diffs to impact categories) and any computed attribution / sensitivity analysis.
  • Verdict sentence / delta-first narrative ordering.
  • Comparison-specific xlsx export (each model keeps its own).
  • Three or more models; copying/moving objects between models; merging models.
  • Persistent server-side model libraries and multi-user collaboration (constitution out-of-scope).

4. UX / interface — the crux of this feature

Chosen layout: the Comparison dashboard — a dedicated full-page view reached from a third top-level tab, alongside the two model tabs. Per-model workspaces stay exactly as today; the dashboard is additive. The two other layouts explored (results-panel mode, delta-first story) were considered and not chosen; variation 2 holds the fuller exploration of this layout, including the deferred Sankey pairing and Δ-only toggle now in backlog.html.

4.4 Time spans — the v1 simplification

v1 computes every aggregate as a full-timespan sum over each model's own modeling period, and plots both time-series on a shared calendar axis. This is exact and needs no normalization. Its honest limitation: when the two models cover different durations, the headline delta compares totals accumulated over different lengths of time — each model's card states its own period so the reader can see this. Normalization for differing timespans (per-year basis, aligned starts) is deferred to backlog.html §2.

5. Constraints

  • Session-driven state remains the persistence model: both models, their names, and the active selection live in the session — no server-side library (constitution.md §4). mission.md currently phrases session state as "the current model lives in the Django session" (singular); that sentence needs a one-line amendment when this ships. Not a constitutional conflict.
  • File-format continuity. Per-model exports keep today's single-model format; files must circulate freely between single-model and two-model sessions, both directions. The combined workspace file is additive — it must not change the single-model format.
  • The library is the domain truth (constitution.md §1.3): the comparison view presents and subtracts results computed by efootprint; it introduces no new modeling logic. Per-category deltas are arithmetic on existing per-category results; no attribution or sensitivity analysis is claimed.
  • HTMX-first, no SPA (constitution.md §1.4) applies to the switcher and the comparison dashboard like everywhere else.
  • Magnitude honesty. Shared scales/legends in the two compared charts (§4.3) are a hard requirement, not a styling preference — independently-scaled side-by-side charts would visually lie.
  • Performance. Session payload and recomputation surface roughly double. Editing one model must not trigger recomputation of the other; switching models and opening the comparison must stay responsive on typical models.
  • Existing flows (onboarding, recovery page, help drawer, xlsx export) keep working unchanged in single-model sessions.

6. Resolved decisions

Settled 2026-06-16 (were open questions in earlier revisions):

  • Decomposition granularity. The "what explains the difference" bars use the full hardware-type taxonomy (servers/network/devices/edge × usage/fabrication), as shown in §4.2 — not a coarser collapse.
  • Differing-timespan rendering. Non-overlapping buckets are left blank; each model card states its own period, so the asymmetry is documented without a greyed marker.
  • Duplicate naming. Duplication proposes "Copy of {system name}", editable by the user afterwards.
  • Assumptions diff matching. Objects are matched by id first, then by (name, type) — robust to renames in the duplicate-then-edit workflow while still diffing independently-built models (§4.2, §4.3).
