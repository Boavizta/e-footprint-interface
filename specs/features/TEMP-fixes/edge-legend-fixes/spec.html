Edge Legend Fixes

Status: Spec — under review · 2026-06-30

1. Problem and audience

Two independent visual issues in the results charts, both affecting any user who reads impact output:

IssuePageProblem
A Edge items always in legend Results — single model The bar chart legend always shows Edge devices — energy and Edge devices — fabrication capsules, even when the edge-modeling toggle is off. These categories have no data and no bars when edge is inactive, so the legend entries are misleading noise.
B Comparison legend wrapping Comparison view — two systems The paired bar comparison chart uses the default Chart.js legend, which wraps items across multiple lines without grouping. In practice, colors for System 1 and System 2 are interleaved across lines, making it hard to identify which color belongs to which system.

Both issues are encountered by any user who reaches the results view. Issue A is the more frequent path (single-model results is the default view); issue B is encountered on every comparison run. Severity is cosmetic but directly undermines trust in chart readability.

2. Success criteria

  • A1. When the edge-modeling toggle is off, no edge-device entries appear in the single-model results bar chart legend.
  • A2. The legend correctly reflects the toggle state when the results chart is rendered (the toggle is not accessible while viewing results, so no live update is required).
  • A3. All non-edge legend entries are unaffected by the toggle state.
  • B1. In the comparison paired bar chart, legend items are left-aligned and visually grouped by system, so the reader can immediately match a color to System 1 or System 2.
  • B2. The comparison legend layout is stable regardless of how many hardware types are active (no mid-list wrap).

3. Scope

Capability-level only. File paths, class/method/parameter names, refactor instructions, internal hypotheses, and "mirror pattern X" code-level decisions belong in plan.html, not here.
In scope
  • Single-model results bar chart: hide edge-device legend items when edge modeling is inactive; show them when active. State is read at chart render time.
  • Comparison paired bar chart: replace the default wrapping legend with a grouped, left-aligned layout — one row per system.
  • Update the view-results and compare-models design-hub journeys (specs/design/journeys/) to document the corrected behaviour.
Out of scope (this iteration)
  • Line (cumulative) chart legend — no edge-toggle interaction needed there today.
  • Comparison cumulative overlay and decomposition charts — only the paired bar chart's legend is addressed.
  • Hiding edge-device datasets (bars, tooltips) — only the legend items are hidden; if datasets carry zero values when edge is off, nothing renders anyway.
  • Any change to the edge-modeling toggle behaviour itself.

5. Constraints

  • Constitution §1.4 — vanilla JS only; no new npm dependency, no SPA framework.
  • Edge-toggle state is a CSS class on <body>; chart-side code reads it from the DOM rather than maintaining a separate copy.
  • Legend rebuilds must not regress chart performance (lightweight DOM patch, no full chart re-creation).
  • Jest test suite must remain green; new behaviour should be covered by new tests.
  • Design-hub journeys are HTML documents following the conventions in specs/design/journey-authoring.md.

6. Open questions

  • B — system labels. ✅ Resolved: row labels use the real system names as entered by the user.
  • A — live update trigger. ✅ Resolved: no live update needed. The toggle is not accessible while viewing results; the legend reads toggle state once at render time.
