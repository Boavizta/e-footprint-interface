Compare view as a resident sibling — Implementation plan

Status: Plan — under review · 2026-06-25 · No spec (technical optimization, no functional change — follows on from model-comparison)

Folder slug is compare-overlay-layer, but “layer” here means an in-flow sibling block toggled by display — not an absolutely-positioned overlay (see §5).

1. Approach

Today the ⇄Compare tab does hx-get /compare/ with hx-target="#main-content-block" hx-swap="innerHTML", which replaces the whole builder — including the two resident canvases. That destruction is the only reason returning to a model needs a full reload: with no resident canvas to reveal, switchToSlot (model_comparison.js) falls back to htmx.ajax("GET","/model_builder/"), which re-hydrates one or two ModelWeb objects from JSON and re-renders the whole builder (the ~1.2 s model_builder_main in production logs). The cost is the bare json_to_system deserialization, not computation (computed attributes are cached).

Make the comparison view a third resident sibling alongside the two canvases, in normal document flow below the tab strip. Opening Compare hides the canvas region (and the model toolbar) with d-none and shows the comparison block; switching to a model hides the comparison block and un-hides that canvas — the exact display toggle the two canvases already use. No canvas is destroyed, so returning to a model is a client-side reveal, not a reload. The comparison fragment is still fetched fresh from /compare/ on every open and emptied (its three Chart.js instances destroyed) on leave, so “no stale results” is unchanged — only the canvases persist.

Because every model tab now reveals a resident canvas (never a canvas-less reload), the skip_chrome branch and the canvas-absent reload branch both disappear, and the comparison view stops carrying its own duplicate tab strip. There is one tab strip in the DOM at all times. Opening Compare hides the ⇄Compare tab and clears the active-model highlight; dismissing restores them — the same client-side display/class toggling a model switch already does. Hiding ⇄Compare while comparing also leaves just the two fixed-width model tabs, so on mobile both fit at once — removing the horizontal scroll the three-tab strip needs today to reach the Reference model.

3. Cross-cutting concerns

  • Mobile dismiss (preserve today’s behavior on the unified strip). A mobile-visible flex-nowrap overflow-x-auto strip already exists today — but it’s the dashboard’s own strip (rendered with on_dashboard=True); the builder strip is d-none d-lg-flex (desktop-only, since mobile switches via the toolbar burger). This refactor deletes that second strip, so the single strip must reproduce the existing behavior: while .comparing, reveal the one strip on mobile (the toolbar/burger is hidden then). No new control — same affordance, carried by a class toggle instead of a separate strip. Improvement: because ⇄Compare is hidden while comparing, the strip is down to two fixed-width model tabs, which both fit on a phone — removing today’s scroll-left-to-reach-Reference.
  • Leader lines. Builder lines live on <body> with their own resize/scroll listeners. On open, call removeAllLines() (the canvas goes d-none); on dismiss, switchToSlot already calls initModelBuilderMain(), which rebuilds the visible canvas’s lines. This reuses the existing parked-canvas line lifecycle — no new mechanism, and because the comparison view is in flow it never overlaps the lines.
  • Unsaved-edits gate (reworked — the resident builder makes opening Compare non-destructive). Because the side panel now survives behind the comparison view, opening Compare discards nothing, so /compare/ leaves PANEL_DISCARDING_PATHS; the discard decision moves to the moment a discard actually happens — a cross-slot /switch-model/ (which stays guarded). Same-slot dismiss is client-side only (no POST), so it can’t trip that guard and falsely warn about losing edits while returning to the model that keeps them. On a cross-slot dismiss the comparison view is torn down first, so the warning is anchored on the edited model, not shown over the dashboard.

4. Risks

  • Chart.js leak on reopen. Emptying the comparison view’s DOM without .destroy() leaves orphaned Chart instances bound to removed canvases. Mitigated by destroyComparisonCharts() on leave.
  • Dismiss-before-warn ordering. Anchoring the unsaved-changes modal on the edited model (not the dashboard) relies on the capture-phase tab-click handler tearing down the comparison view before htmx’s bubble-phase request fires its htmx:confirm guard. Verify the modal lands over the revealed model, that “Continue” still re-fires the exact /switch-model/ request (issueRequest(true)), and that “Cancel” leaves the user anchored on that model (comparison already dismissed). (The earlier “stale panel” risk is gone: same-slot reappearance is the same model — desired — and cross-slot is closed by the switch teardown.)
  • Memory: builder + comparison resident together. Both canvases plus the ~95 KB comparison fragment + 3 charts coexist while Compare is open. Acceptable (the fragment is emptied on leave), but note it.
  • Flow/layout regression. The builder uses flexbox with no height variables (flex: 1 1 0; min-height: 0;). The comparison view must drop into the same flex parent and scroll internally (#comparison-dashboard already overflow-y-auto) without reintroducing height vars. Low risk — verify the toolbar+canvas d-none reclaims space cleanly.

5. Alternatives considered

  • Absolutely-positioned overlay floating over the canvas. Rejected (earlier draft of this plan). A floating layer stacks against the leader-line SVGs on <body>, which the “No clipping container” rule keeps off the chrome only via opaque, higher-z bars (navbar z-2, toolbar z-3, strip z-4) — forcing a z-index threaded between z-3 and z-4, and a separate mobile-dismiss affordance. The in-flow sibling avoids both entirely: real content with real height overlaps nothing.
  • Move the toggle-vs-reload decision into switch_model (server HX-Location). Rejected in discussion: couples a state-write endpoint to layout ids it can’t observe, and splits the “react to switch” logic across two mechanisms. Doesn’t address the root cost (canvas destruction).
  • Keep the comparison view’s own self-contained tab strip and just stop destroying the builder. Rejected: two #model-tab-strip ids in the DOM at once (the switch JS depends on getElementById). The single-strip model is cleaner and removes on_dashboard entirely.
  • Keep canvases but re-fetch chrome only. Rejected: doesn’t remove the reload; the win comes specifically from not destroying the canvases.

6. Constitutional notes

  • §1.4 HTMX-first; minimize JS. Roughly JS-neutral: deletes the canvas-absent reload branch and the skip_chrome path, adds an openCompareView/dismissCompareView pair and a capture-phase tab-click handler — all small display toggles in the same vein as the existing model-active toggle in switchToSlot. No new client-side routing or data layer.
  • §1.3 Library is domain truth. Untouched — the comparison is still computed by System.compare_to via ComparisonService; only the render container changes.
  • §2 Quality gates. No Django-model change (no migration). CHANGELOG entry required; pytest + Jest green; pyproject.toml back on PyPI efootprint before merge.
  • No §4 rule relaxed; no constitutional amendment needed.

7. Decisions (resolved)

  • Toolbar while comparing → hidden (d-none). Keeps the comparison self-contained (no per-model chrome over a two-model view), restored on dismiss. One display toggle.
  • Hidden canvas region → d-none. Dismiss becomes a pure reveal; leader lines are torn down on open and rebuilt on dismiss regardless.
  • {% elif dashboard %} branch in base.html → removed. Compare is only ever an in-flow sibling swap, so the full-page dashboard render is unreachable. Delete the branch and confirm no view passes dashboard=True to a full-page render.
