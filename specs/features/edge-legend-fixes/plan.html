Edge Legend Fixes — Implementation plan

Status: Plan — under review · 2026-06-30 · Spec: spec.html

1. Approach

A Edge items in single-model bar chart

The splitCapsuleLegendPlugin in legend.js rebuilds the legend from chart.legend.legendItems on every update. The items are derived from the chart's datasets, which are built once from HARDWARE_TYPE_CONFIG in config.js. The edge toggle state lives on document.body as the CSS class edge-modeling-on / edge-modeling-off.

The fix has two parts: (1) mark edge hardware types in HARDWARE_TYPE_CONFIG with isEdge: true so the flag travels through createDatasetConfig onto the Chart.js dataset object, and (2) in the plugin's rebuild pass, skip any legend item whose dataset carries isEdge: true when body.classList does not contain edge-modeling-on. No chart re-creation is needed — the legend container is cleared and rebuilt on every _legendNeedsRebuild pass anyway.

B Comparison paired bar chart legend

The paired chart payload from ComparisonService already carries per-dataset stack values ("A" or "B"), and dataset labels of the form "{system_name} usage" / "{system_name} fabrication". However, there is no top-level field for the raw system names, so a clean row label requires either string surgery on a dataset label or a new payload field.

We add modelAName and modelBName string fields to the _paired_chart_payload dict in comparison_service.py. On the JS side, we replace the Chart.js built-in legend (legend: { display: true }) with a custom HTML legend rendered into a <div> below the canvas after the chart is created. The custom legend groups datasets by stack, renders one row per system, and uses payload.modelAName / payload.modelBName as row labels. The same treatment is applied to the cumulative overlay chart (renderCumulativeLegend), using a horizontal line swatch to match the line chart visual.

3. Implementation notes

A Edge flag propagation

createDatasetConfig in chart.js spreads HARDWARE_TYPE_CONFIG[hardwareType] directly onto the Chart.js dataset object — so adding isEdge: true to the config entry is enough; the property will be present on chart.data.datasets[i] without any change to chart.js.

In splitCapsuleLegendPlugin.afterUpdate(), the filter runs before the usage/fabrication split: 

const edgeOff = !document.body.classList.contains('edge-modeling-on');
const visibleItems = legend.legendItems.filter(item =>
    !edgeOff || !chart.data.datasets[item.datasetIndex]?.isEdge
);

Then pass visibleItems to filterItemsByCategory instead of legend.legendItems.

Edge case: if the body has neither class (e.g. on first load before the toggle JS runs), treat edge as visible (fail-open). The condition above handles this: !body.classList.contains('edge-modeling-on') is true in that state, so edge items are hidden — which is the correct default since the toggle defaults to off when no preference is stored. Verify this matches edge_modeling_toggle.js behaviour before shipping.

B Custom paired legend

renderPairedLegend(payload, canvasId) is a plain DOM function (no Chart.js dependency) that:

  1. Finds or creates a <div id="{canvasId}-legend"> after the canvas.
  2. Groups payload.datasets by stack ("A" / "B").
  3. For each group, renders one row: a system-name label (payload.modelAName or payload.modelBName) followed by color-dot + dataset-label items.

The row label element reuses the same font/color tokens already established in SHARED_FONT and TICK_COLOR to stay visually consistent. No interaction (no "only" button) — the comparison legend is display-only.

destroyPairedLegend() removes the <div> by id if it exists. It is called from destroyComparisonCharts() so cleanup stays symmetric.

Add a Python unit test in tests/unit_tests/ asserting modelAName and modelBName are present in the paired payload.
  • Migrations: No Django model change. No session schema change (chart payload is computed on the fly from the session system data).
  • Docs: specs/architecture.md mentions ComparisonService — add a one-line note that the paired payload now includes system name fields. specs/conventions.md may need a note on the isEdge pattern if it becomes a general precedent.
  • Build: JS changes are in the result_charts bundle; rebuild with npm run build:result-charts:dev (or the prod equivalent) before testing.

5. Risks

  • Edge-off default state on first load. If body has neither class before the toggle JS runs, edge items will be hidden by the fail-open condition. Verify timing: the toggle script runs on DOMContentLoaded and the chart is drawn after triggerResultRendering (an HTMX event), so the body class should be set before any chart renders. Confirm in an E2E smoke test.
  • Payload backward compatibility. Adding modelAName / modelBName to the paired payload is additive (JSON; the JS reads these new fields). Old cached payloads (stored as data-attributes on the dashboard element) will lack the fields — renderPairedLegend should fall back gracefully (e.g. "System A" / "System B") when the fields are absent.
  • Legend container ID collision. renderPairedLegend shares the {canvasId}-legend naming pattern with the single-model legend plugin. Since the two are on different pages/tabs this is not a runtime collision, but the convention should be noted so future work doesn't accidentally reuse the same id on one page.

6. Alternatives considered

  • A — CSS hide via body class. Hide edge capsules with a CSS rule targeting body.edge-modeling-off [data-is-edge]. Rejected: the custom legend is built dynamically with inline styles; a data-attribute approach would work but mixes display logic across CSS and JS. The plugin approach keeps all legend logic in JS.
  • A — Re-create the chart on toggle. Trigger a full chart redraw when the edge toggle changes. Rejected: over-engineered (the toggle is inaccessible while viewing results, per spec), and expensive.
  • B — Extract system name from dataset label string. Strip " usage" / " fabrication" suffix from the first stack-A dataset label. Rejected: brittle — depends on label format staying stable; the backend field is the authoritative source.
  • B — Keep Chart.js built-in legend, tune wrapping. Set legend.maxWidth or style the legend box to wrap at system boundaries. Rejected: Chart.js doesn't support semantic grouping in its built-in legend; achieving it would require overriding the label-generation callback, which is harder to test and less readable than a simple DOM render.

7. Constitutional notes

All changes are vanilla JS (frontend) or pure Python (service layer). No new npm dependency, no framework, no Django model change. Consistent with constitution §1.4 (HTMX-first, minimize JS) and §1.3 (library is domain truth — ComparisonService wraps library output, does not re-implement it). No constitutional deviation.
