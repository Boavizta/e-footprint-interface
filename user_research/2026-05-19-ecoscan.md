# EcoScan — e-footprint modeling exercise

- **Date:** 2026-05-19
- **Source:** External user modeling the EcoScan monorepo (`apps/web`, `apps/engine-api`,
  Supabase) with e-footprint v21.x + JSON export. Wrote up a structured feedback document.
- **Disposition:** Most "limitations" were **discoverability** failures (the capability existed
  but wasn't found), one was a genuine modeling-convenience gap (managed databases), and a few
  are real roadmap/DX items.

## What they valued

Clear object graph (UsageJourney → Step → Job → Server), `BoaviztaCloudServer` defaults,
explainability (SourceValue + Sankey), JSON round-trip + GUI editing, and the docs/tutorial.

## Limitations they reported, and our reading

| They reported | Our reading | Disposition |
|---|---|---|
| Serverless/PaaS not modeled; proxied Vercel as a low-utilization VM. | `ServerTypes.serverless()` already models fractional, pay-per-use instances; `Job` is already per-invocation. **Discoverability gap.** | Addressed → `saas-serverless-modeling-ergonomics` (how-to). |
| Managed BaaS (Supabase/Neon/RDS) not first-class. | A database is server + storage + jobs; "managed" emerges from a serverless host. Energy-given-resources is already solid; the hard part (query→resource cost) has no defensible general model. | Addressed **docs-only** (how-to). Query-cost modeling logic parked → e-footprint `specs/modeling_logic_roadmap.md`. |
| Need 8760 hourly values / hand-authoring JSON is error-prone. | `time_builders` exist and are already shown in the tutorial; **no 8760 enforcement** (a 1-hour series is valid; period inferred from data). The interface-editable builders are weak, though. | Discoverability already covered (tutorial). Interface-editable unified builders **parked** → interface roadmap. |
| Multi-tenant SaaS: per-tenant allocation not built-in. | Per-usage-pattern attribution is fully built in and covers all tiers (devices, network, server, storage) — one UP per tenant gives a complete per-tenant footprint; serverless covers the shared-platform fraction. | Addressed (docs how-to). Only gap: attributing network/devices to a *provider* (low priority, see backlog). |
| Boavizta API called at import time. | Confirmed: N+1 HTTP calls at import; in-memory cache only. Real wart. | Addressed → Boavizta import-time fix track. |
| Python 3.12+; install failed on 3.14 (native deps). | Upstream wheel-availability issue, not a hard cap. Needs a support-window policy + CI. | Addressed → packaging/DX track. |
| No templates for third-party integrations (OAuth workers, webhooks, PDF). | Likely expressible as Jobs today; unclear if any primitive is missing. | **Parked**, clarification pending (see backlog). |
| Carbon only; wants the 9 PEF impacts. | Multicriteria needs a new multi-impact primitive. | **Roadmap.** |

## Their forward suggestions (P1–P6) → where they land

- **P1 SaaS/serverless archetypes + per-request job:** mostly via docs, not new classes. A
  database how-to yes; the `Database` *primitive* was descoped and its query-cost modeling logic
  parked (no defensible general model — see e-footprint `specs/modeling_logic_roadmap.md`). A
  serverless-function *archetype* was **deliberately rejected** (CPU load is the impact driver and
  must stay an explicit input, so an archetype that derives CPU from memory would mislead).
- **P2 CLI + scenario file + minimal-series docs:** timeseries already shown in the tutorial;
  interface-editable builder unification **parked** → interface roadmap. CLI/scenario file **parked**.
- **P3 operational importers + telemetry adapter:** **parked**.
- **P4 wheels / Docker / minimal extra:** taken → feature `packaging-and-dx` (support window + CI + Docker). The "minimal extra" idea (optional viz/pandas) parked on the memory-optimization track; zstd kept (benchmark).
- **P5 GHG/ISO reporting block + per-transaction metrics:** **parked** (reporting export and
  first-class per-tenant/per-transaction accessor).
- **P6 EcoScan VSME B3 bridge:** EcoScan-specific; not an e-footprint deliverable.
